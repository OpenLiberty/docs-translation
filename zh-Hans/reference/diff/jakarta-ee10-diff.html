Jakarta EE 10 与 9.1 之间的差异

如果要将应用程序从使用 Jakarta EE 9.1 功能部件更新为使用 Jakarta EE 10 功能部件，那么 API 行为的更改可能需要您更新应用程序代码。 这些更改适用于多个 Jakarta EE API 以及支持这些 API 的 Open Liberty 功能部件。

以下部分描述了不同 Jakarta EE 10 功能部件的更新

  • Jakarta EE 10 与 9.1

  • Jakarta Authentication 3.0

  • Jakarta Authorization 2.1

  • Jakarta Batch 2.1

  • Jakarta Contexts and Dependency Injection 4.0

  • Jakarta Expression Language 5.0

  • Jakarta Faces 4.0

  • Jakarta Mail 2.1

  • Jakarta RESTful Web Service 3.1

  • Jakarta Security 3.0

  • Jakarta Server Pages 3.1

  • Jakarta Servlet 6.0

  • Jakarta WebSocket 2.1

  • Jakarta XML Binding 4.0

  • Jakarta XML Web Services 4.0

Jakarta EE 10 与 9.1

以下更改适用于 Jakarta EE 10 平台。

Java SE 需求

Jakarta EE 9.1 功能部件可以与 Java SE 版本 8 和更高版本一起运行。 所有 Jakarta EE 10 功能部件都需要 Java 11 或更高版本，如 Jakarta EE 10 规范所概述。

新建核心概要文件

Jakarta EE 10 引入了名为 Core Profile 的新概要文件。 提供核心概要文件功能的 Liberty 功能部件包括 Jakarta Contexts and Dependency Injection 4.0 (cdi-4.0) ， Jakarta JSON Binding 3.0 (jsonb-3.0) ， Jakarta JSON Processing 2.1 (jsonp-2.1) 和 Jakarta RESTful Web Services 3.1 (restfulWS-3.1)。 没有为 "核心概要文件" 提供方便功能。 但是，您可以通过在 server.xml 文件中同时包含 restfulWS-3.1 (包括 cdi-4.0 和 jsonp-2.1) 和 jsonb-3.0 Liberty 功能部件，轻松启用所有 Jakarta 10 Core Profile 功能部件。

XML 绑定 API 可用性

若干 Jakarta EE 9.1 Liberty 功能部件包含对 Jakarta XML Binding 3.0 (xmlBinding-3.0) 功能部件的依赖关系，即使规范未要求这些功能部件依赖于或公开 XML 绑定 API 也是如此。 此外，其中许多功能部件是 Jakarta Web Profile 功能部件子集的一部分，它不包含 XML 绑定功能部件。

以下 Jakarta EE 10 Liberty 功能部件不会公开 XML 绑定 4.0 API ，其中其等效的 Jakarta EE 9.1 功能部件会公开 XML 绑定 3.0 API:

  • Jakarta Authentication 3.0 (appAuthentication-3.0)

  • Jakarta Connectors 2.1 (connectors-2.1)

  • Jakarta Persistence 3.1 (persistence-3.1)

  • Jakarta 持久性容器 3.1 (persistenceContainer-3.1)

  • Jakarta EE Web 概要文件 10.0 (webProfile-10.0)

对于需要 XML 绑定 API 的 Open Liberty 应用程序，现在必须将 Jakarta XML Binding 4.0 (xmlBinding-4.0) Liberty 功能部件添加到 server.xml 文件。

管理 REST 连接器可用性

Jakarta EE Platform 9.1 (jakartaee-9.1) Liberty 便利功能部件包含 管理 REST 连接器 2.0 (restConnector-2.0) Liberty 功能部件。 Jakarta EE Platform 10.0 (jakartaee-10.0) Liberty 便利功能部件不包含此 Liberty 功能部件。

如果使用 jakartaee-10.0 方便功能部件，并且应用程序需要 "管理 REST 连接器" 功能，那么必须将 restConnector-2.0 功能部件添加到 server.xml 文件。

连接器入站安全性更换

在 Jakarta EE 10 中， Jakarta Connectors 2.1 (connectors-2.1) 和 Application Security 5.0 (appSecurity-5.0) Liberty 功能部件的组合将替换 Jakarta EE 9.1中提供的 Jakarta Connectors 入站安全性 2.0 (connectorsInboundSecurity-2.0) Liberty 功能部件。

JSON Web 令牌 (JWT) 功能部件更改

当您使用 Jakarta EE 10 Liberty 功能部件启用 JSON Web 令牌 1.0 (jwt-1.0) Liberty 功能部件时，缺省情况下 Servlet API 不再可用。 对于需要 Servlet API 的 Open Liberty 应用程序，必须将 Jakarta Servlet 6.0 (servlet-6.0) 功能部件添加到 server.xml 文件。

同样，当使用 Jakarta EE 10 Liberty 功能部件启用 jwt-1.0 功能部件时， com.ibm.wsspi.rest.handler， com.ibm.wsspi.rest.handler.helper和 com.ibm.wsspi.wab.configure SPI 不再可用于用户功能部件。 对于需要这些 SPI 的 Open Liberty 用户功能部件，必须将 com.ibm.websphere.appserver.restHandler-1.0 或 com.ibm.wsspi.appserver.webBundle-1.0 功能部件添加到用户功能部件 mf 文件。

Jakarta Authentication 3.0 与 2.0 之间的差异

Jakarta Authentication 3.0 功能部件 (appAuthentication-3.0) 引入了对 jakarta.security.auth.message.config.AuthConfigFactory 抽象类的更改。 此类实现以下两种新方法来注册和除去服务器认证模块。

  • public abstract String registerServerAuthModule(ServerAuthModule serverAuthModule, Object context);

  • public abstract void removeServerAuthModule(Object context);

在先前版本的功能部件中，支持仅注册和除去配置提供程序。

有关更多信息，请参阅 Jakarta Authentication 3.0 Javadoc。

Jakarta Authorization 2.1 与 2.0 之间的差异

Jakarta Authorization 2.1 功能部件 (appAuthorization-2.1) 引入了以下新方法。

jakarta.security.jacc.PolicyConfigurationFactory 抽象类的新方法

jakarta.security.jacc.PolicyConfigurationFactory 抽象类实现以下两种新方法来检索策略配置接口 (PolicyConfiguration):

  • public abstract PolicyConfiguration getPolicyConfiguration();

  • public abstract PolicyConfiguration getPolicyConfiguration(String contextID);

除了先前版本中提供的以下 getPolicyConfiguration 方法外，还提供了这些方法。

public abstract PolicyConfiguration getPolicyConfiguration(String contextID, boolean remove);

jakarta.security.jacc.PolicyConfiguration 接口的新方法

jakarta.security.jacc.PolicyConfiguration 接口定义了三种新的读许可权方法:

  • public abstract Map<String, PermissionCollection> getPerRolePermissions();

  • public abstract PermissionCollection getUncheckedPermissions();

  • public abstract PermissionCollection getExcludedPermissions();

有关更多信息，请参阅 Jakarta Authorization 2.1 Javadoc。

Jakarta Batch 2.1 与 2.0 之间的差异

Jakarta Batch 2.1 中的更新主要侧重于改进与上下文依赖关系注入 (CDI) 的集成。 现在，批处理工件的发现已使用 CDI 进行标准化，先前未在规范中完全定义 CDI。

@BatchProperty 注释的更新

展开 @BatchProperty 注释以启用以下类型的注入: Boolean， Short， Integer， Long， Float和 Double。 先前，此注释只能注入 String 值。 以下示例演示了新添加的注入类型:

@Inject @BatchProperty(name="b1") Boolean b1;
@Inject @BatchProperty(name="s1") Short s1;
@Inject @BatchProperty(name="i1") Integer i1;
@Inject @BatchProperty(name="l1") Long l1;
@Inject @BatchProperty(name="f1") Float f1;
@Inject @BatchProperty(name="d1") Double d1;

JobOperator 接口的缺省注入

先前，仅通过 BatchRuntime.getJobOperator() 静态工厂方法提供了缺省 JobOperator 接口。 在 Jakarta Batch 2.1 及更高版本中，如果存在 JobOperator 接口的注入点，并且未找到用户提供的实现，那么将自动注入缺省 JobOperator 实例。 以下示例演示了 JobOperator 接口的缺省注入:

@Inject JobOperator jobOperator;

Jakarta Contexts and Dependency Injection 4.0 与 3.0 之间的差异

Jakarta Contexts and Dependency Injection 4.0 功能部件 (cdi-4.0) 引入了构建兼容扩展和可观察的容器状态事件。 它还会更改处理空 beans.xml 文件和未版本化文件的方式。 在此发行版中除去了一些先前不推荐使用的 API。

构建兼容扩展

CDI 提供可移植扩展以定制 CDI 应用程序初始化生命周期。 在 CDI 4.0中，构建兼容扩展使实现扩展能够进行构建时处理。

要实现构建兼容扩展，请提供 META-INF/services 目录中声明的 BuildCompatibleExtension 接口的实现。 该实现可以提供使用下列其中一个扩展注释进行注释的方法，每个注释对应于一个扩展执行阶段:

  • @Discovery

  • @Enhancement

  • @Registration

  • @Synthesis

  • @Validation

例如，作为 "增强" 阶段的一部分，以下示例中的实现将 MyQualifier 注释添加到 MyService 类型:

public class MyExtension implements BuildCompatibleExtension {
    @Enhancement(type=MyService.class)
    public void addMyQualifier(ClassConfig clazz) {
        clazz.addAnnotation(MyQualifier.class)
    }
}

启动和关闭事件

在 CDI 4.0: Startup 和 Shutdown。 应用程序可以侦听要在 CDI 容器启动或关闭时通知的这些事件。 以下示例侦听 Startup 和 Shutdown 事件，并在控制台接收到事件时将每个事件的通知打印到控制台:

@ApplicationScoped
public class MyObserver {
    public void observeStartup(@Observes Startup startupEvent) { {
        System.out.println("CDI Container is starting");
    }

    public void observeShutdown(@Observes Shutdown shutdownEvent) { {
        System.out.println("CDI Container is stopping");
    }
}

您可以使用 @Priority 注释来控制多个可观察容器状态方法之间的顺序。

空和非空 beans.xml 文件中的 bean-discovery-mode 属性

在 CDI 4.0中，未指定 bean-discovery-mode 属性的 beans.xml 文件将被视为属性设置为 bean-discovery-mode="annotated"。 在 CDI 3.0 和更低版本中，未指定 bean-discovery-mode 属性的 beans.xml 文件将被视为属性设置为 bean-discovery-mode="all"。 这些设置适用于空和非空 beans.xml 文件。

对于 CDI 4.0中的空 beans.xml 文件，可以设置 cdi 元素的 emptyBeansXmlCDI3Compatibility 属性以实现与先前版本的兼容性。 如果在 server.xml 文件中将此属性设置为 true ，那么包含空 beans.xml 文件的任何归档都将被视为显式 Bean 归档。 有关更多信息，请参阅 Server.xml 配置元素和属性更新。

最佳实践是始终为非空 beans.xml 文件指定版本，如以下示例中所示:

<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xmlns="https://jakarta.ee/xml/ns/jakartaee"
    xsi:schemaLocation="https://jakarta.ee/xml/ns/jakartaee https://jakarta.ee/xml/ns/jakartaee/beans_4_0.xsd"
    version="4.0"
    bean-discovery-mode="annotated">
</beans>

Bean 和实例的程序化查找

在 CDI 4.0中，提供了新的 Handle API 来简化 Bean 元数据的程序化检查。 此 API 避免了在需要实例之前创建这些实例的需要。 您可以使用 Instance API 获取 Handle 实例，该 API 使用以下语法:

public interface Handle<T> extends AutoCloseable {
        T get();
        Bean<T> getBean();
        void destroy();
        void close();
}

已除去 API

在 CDI 4.0:

  • @New 限定符，由 @Dependent Bean 替换。

  • Bean#isNullable()，未由自 CDI 1.1以来的实现使用。

  • BeanManager#createInjectionTarget(AnnotatedType)，替换为 BeanManager#getInjectionTargetFactory(AnnotatedType)

  • BeforeBeanDiscovery#addAnnotatedType(AnnotatedType)，替换为 BeforeBeanDiscovery#addAnnotatedType(AnnotatedType, String)

Server.xml 配置元素和属性更新

在 Liberty Jakarta 上下文和依赖关系注入 4.0 功能部件中， cdi12 配置元素由适用于 CDI 版本 1.2 和更高版本的 奇迪 元素取代。 以下 server.xml 文件示例显示了具有两个配置属性的 cdi 元素:

<cdi enableImplicitBeanArchives="false" emptyBeansXmlCDI3Compatibility="true"/>

enableImplicitBeanArchives 属性的工作方式与先前版本中的 cdi12 元素相同。 如果此属性设置为 true(缺省值) ，那么没有 beans.xml 文件的归档将被视为隐式 Bean 归档，并扫描以查找具有 Bean 定义注释的类。 如果此属性设置为 false，那么不会扫描不包含 beans.xml 文件的归档以查找带注释的类。

Jakarta Servlet 6.0 与 5.0 之间的差异

Jakarta Servlet 6.0 功能部件 (servlet-6.0) 提供了新的 API 和新的配置来设置响应 cookie 上的属性。

配置 cookie 属性的新方法

在 Jakarta Servlet 6.0中，可以从部署描述符文件，使用 jakarta.servlet.ServletContainerInitializer API 在应用程序启动时或在请求时设置响应 cookie 上的属性。

要在 web.xml 部署描述符文件中设置 cookie 属性，请在 cookie-config 元素中指定 attribute 元素，如以下示例中所示:

<session-config>
   <cookie-config>
     <name>SampleCookieName</name>
     <domain>my.domain.com</domain>
     <path>/</path>
     <http-only>true</http-only>
     <secure>true</secure>
     <attribute>
        <attribute-name>MyAttributeName</attribute-name>
        <attribute-value>valueExample</attribute-value>
     </attribute>
     <attribute>
        <attribute-name>MyOtherAttribute</attribute-name>
        <attribute-value>OtherValue</attribute-value>
     </attribute>
   </cookie-config>
</session-config>

您还可以在应用程序启动期间使用 jakarta.servlet.ServletContainerInitializer API 从应用程序动态设置 cookie 属性，如以下示例中所示:

@Override
public void onStartup(Set<Class<?>> setOfClasses, ServletContext context) throws ServletException {
      SessionCookieConfig scc = context.getSessionCookieConfig();
      scc.setName("MyDynaCookieName");
      scc.setAttribute("MyDynamicAttribute", "DynamicValue");
      ...
}

或者，您可以在请求时从应用程序设置属性，如以下示例中所示:

SessionCookieConfig scc = getServletContext().getSessionCookieConfig();
scc.setName("MyRequestCookieName");
scc.setAttribute("MyRequestAttribute", "RequestValue");

如果 cookie 配置的不同源之间发生冲突，那么以下优先顺序适用于解决冲突，从最高优先顺序到最低优先顺序:

  1. 应用程序在请求时设置的属性

  2. 应用程序使用 ServletContainerInitializer API 设置的属性

  3. web.xml 部署描述符文件中设置的属性

对请求 Cookie 头的更改

Servlet 6.0 通过遵循 rfc6265 标准来实现 HTTP 请求 cookie 行为，该标准指出 HTTP 客户机代理程序 (例如浏览器) 不会将 cookie 属性发送回应用程序服务器。

先前，可以通过将美元符号 ($) 作为前缀添加到有效属性名称来包含请求 cookie 属性。 例如， $Domain=myDomain.com 被视为请求 cookie 的 Domain 属性。 从 Servlet 6.0开始， $Domain 将被视为名为 $Domain的新 cookie。 美元符号也是 cookie 名称的一部分。 此规则的唯一例外是 $Version 值。

例如，请考虑以下 HTTP 请求，其中包含 cookie 头:

Cookie: $Version=1; name1=value1; $Path=/Dollar_Path; $Domain = localhost; $NAME2=DollarNameValue;Domain=DomainValue

在 Servlet 6.0之前，服务器从此请求创建了以下三个请求 cookie:

cookie1: name1 ，具有关联的属性 :Path=/Dollar_Path; Domain=localhost
cookie2: NAME2
cookie3: 域

在 Servlet 6.0 和更高版本中，服务器从同一 HTTP 请求创建以下五个请求 cookie:

cookie1: name1
cookie2: $Path
cookie3: $Domain
cookie4: $NAME2
cookie5: 域

应用程序调试增强功能

新的 jakarta.servlet.ServletRequest API 和新的 jakarta.servlet.ServletConnection 类可用于帮助您调试对应用程序的请求。 通过增强功能，您可以跟踪或引用请求，或者使用请求标识获取请求所使用的网络连接的详细信息。

以下 Jakarta.servlet.ServletRequest API 在 Servlet 6.0:

  • String getRequestId()

  • String getProtocolRequestId()

  • ServletConnection getServletConnection()

将从 getServletConnection() servlet 请求中检索新的 jakarta.servlet.ServletConnection 对象。 它包含以下 API:

  • String getConnectionId()

  • String getProtocol()

  • String getProtocolConnectionId()

  • boolean isSecure()

URI 路径验证

从 Servlet 6.0 发行版开始，如果 URI 中存在以下任何序列，那么将拒绝具有 400 错误请求状态的直接请求:

  • 编码的点 (%2E) 字符

  • 编码的正斜杠 (%2F) 字符

  • 片段 (#) 字符，无论是否已编码 (%23)

  • 反斜杠字符 (\) ，无论是否已编码 (%5C)

  • 任何控制字符，无论是否已编码

  • 具有路径参数 (;) 的任何句点 (.) 或双句点 (..) 段，例如 /path/..;/info

  • 以双句点 (..) 开头的任何路径，例如 /../path/info

注: URI 路径验证未应用于查询字符串。

您可以通过设置 skipEncodedCharVerification 属性 true来跳过对编码字符的验证。 缺省值为 false。

要选择退出 servlet 的请求 URI 中的编码字符验证，可以将此属性配置为应用程序 web.xml 文件中的 context-param 元素。

<context-param>
   <param-name>SKIP_ENCODED_CHAR_VERIFICATION</param-name>
   <param-value>true</param-value>
</context-param>

或者，您可以将其设置为 server.xml 文件中 webContainer 元素的属性，以跳过所有已部署应用程序的验证。

<webContainer skipEncodedCharVerification="true"/>

应用程序级别 context-param 设置优先于服务器级别 webContainer 设置。

X-Powered-By 响应头

缺省情况下，响应头中不包含 X-Powered-By 头。 从 Servlet 6.0开始，没有可用于将 X-Powered-By 头添加到响应头的配置选项。

Jakarta WebSocket 2.1 与 2.0 之间的差异

Jakarta WebSocket 2.1 功能部件实现对规范 API 和方法的更改。 它还包括对用户属性的处理和客户机 SSL 上下文设置的更新。

WsWsocServerContainer#upgrade SPI 的 API 替换

在 Jakarta WebSocket 2.1中，您可以使用新的 jakarta.websocket.server.ServerContainer#upgradeHttpToWebSocket () API 方法，以编程方式将 HTTP 请求升级到 WebSocket 连接。 此方法会将现在不推荐使用的 com.ibm.websphere.wsoc.WsWsocServerContainer SPI 替换为计划在下一个 WebSocket 发行版中将其除去。 但是，此 SPI 仍随 Jakarta WebSocket 功能部件版本 2.0 和更低版本一起提供。

用户属性处理的更新

WebSocket 2.1 发行版包含对 用户属性的更改。 服务器会话的用户属性最初由 ServerEndpointConfig.getUserProperties() 方法填充。 此方法使端点能够检索可能在入局请求的 ServerEndpointConfig.Configurator#modifyHandshake 调用中设置的属性。

同样，将使用 EndpointConfig.getUserProperties() 方法填充 客户机会话的用户属性 。 这些初始内容是浅层副本，这意味着现在将根据每个端点 (每个 WebSocket 会话) 来处理 ClientEndpointConfig#getUserProperties 和 ServerEndpointConfig#getUserProperties() 实例。 所有端点实例之间的属性不再是先前 WebSocket 版本中的属性的全局属性。

客户机 SSL 配置

从 WebSocket 2.1开始，应用程序可以使用 Liberty 用于与服务器建立 wss 连接的 jakarta.websocket.ClientEndpointConfig.Builder#sslContext 实例来设置其 SSLContext 类。 此选项覆盖所有 Liberty 服务器 SSL 配置，必须谨慎使用。 建议通过启用 传输安全性 功能部件在 server.xml 文件中启用 SSL 通信。 此外，您可以通过在 server.xml 文件中设置 WAS WebSocket Outbound 元素来配置 SSL。

其他更改

以下更改从 WebSocket 2.1开始生效。

  • 现在， getter 方法可用于缺省平台配置程序: jakarta.websocket.server.ServerEndpointConfig.Configurator#getContainerDefaultConfigurator()。

  • 将除去在应用程序部署期间必须添加端点的限制。 从 V 2.1开始，可以随时添加新端点。

  • jakarta.websocket.Session.getRequestURI() 方法返回从协议到查询字符串的完整 URI 字符串 (如果该字符串存在)。

有关更多信息，请参阅 Jakarta WebSocket 2.1 规范。

Jakarta XML Binding 4.0 与 3.0

Jakarta XML Binding 4.0 功能部件使您可以选择切换到第三方实现并除去对 Validator API 的支持。

切换到第三方实现

在 Jakarta XML Binding 4.0中，可以指定 jakarta.xml.bind.JAXBContextFactory 属性以切换到首选第三方实现。 然后， xmlBinding-4.0 功能部件使用指定的实现而不是引用实现。 选择第三方实现时，请考虑以下注意事项。

如果设置 jakarta.xml.bind.JAXBContextFactory 属性，那么必须将指定的第三方实现添加到应用程序类路径。 否则，将发生 java.lang.ClassNotFoundException 异常。 此外，必须在 JVM 上运行的所有应用程序之间共享第三方实现。 在未将实现添加到其类路径的 JVM 上运行的任何应用程序都会迂到 java.lang.ClassNotFoundException 异常。

除去验证器 API

xmlBinding-4.0 功能部件不支持 jakarta.xml.bind.Validator API。 使用此 API 的任何应用程序都必须将其除去或将其替换为 SchemaValidation 实例。 以下示例演示如何使用 SchemaValidation 实例作为验证器 API 的替代项:

SchemaFactoryschemaFactory= SchemaFactory.newInstance(XMLConstants.W3C_XML_SCHEMA_NS_URI);
Schema dataObjectSchema= schemaFactory.newSchema(new File("dataObjectSchema.xml")); jaxbUnmarshaller.setSchema(dataObjectSchema)

有关更多信息，请参阅 Jakarta XML Binding 4.0 规范。

Jakarta XML Web Services 4.0 与 3.0 之间的差异

Jakarta XML Web Service 4.0 功能部件包含全局处理程序和 Jakarata Web Service 元数据规范的更改。 它还会除去先前不推荐使用的带有附件的 SOAP 3.0 API 类，该 API 在 Open Liberty 中受 Jakarta XML Web Services 功能部件支持。

  • 通过 4.0 发行版，将 Jakarata Web Service 元数据规范合并到 Jakarta Web Service 规范中。 现在， Open Liberty 的所有 Jakarta Web Service API 都包含在 dev\api\spec\io.openliberty.jakarta.xmlWS.4.0…​jar 文件中。

  • 在 Jakarta XML Web Service 4.0中，以及在 Jakarta EE 10 平台上，缺省情况下会禁用全局处理程序。 有关更多信息，请参阅 全局处理程序。

  • 在 Open Liberty中，带附件的 SOAP 3.0 发行版与 Jakarta XML Web Services 4.0 发行版一致。 通过此发行版，将除去先前不推荐使用的带附件的 SOAP 3.0 API 的 SOAPElementFactory 类。
