服务器配置概述

Open Liberty 服务器配置由一个必需文件， server.xml 文件和一组可选文件组成。 server.xml 文件必须是格式正确的 XML ，并且根元素必须是 server。 处理 server.xml 文件时，将忽略无法理解的任何元素或属性。

此示例 server.xml 文件将服务器配置为执行以下操作:

<server description="new server">
    <featureManager>
        <feature>jsp-2.3</feature>
    </featureManager>
    <httpEndpoint id="defaultHttpEndpoint"
                  httpPort="9080"
                  httpsPort="9443" />
    <applicationManager autoExpand="true" />
</server>

  • feature 元素配置 Open Liberty 以支持 JavaServer Pages 2.3 (jsp-2.3) 功能部件。

  • httpPort 属性设置为 9080，这将配置 Open Liberty 以侦听到端口 9080上的 localhost 的入局流量。

  • autoExpand 属性设置为 true ，以便在部署 WAR 文件时自动展开这些文件。

术语 服务器配置 可用于引用构成服务器配置的所有文件，或者专门引用 XML 文件中的配置。 如果在上下文中不清楚，那么可以使用术语 服务器 XML 配置 来引用 XML 文件中的配置。

配置文件

按以下顺序处理服务器配置文件:

  1. server.env -此文件中指定了环境变量。

  2. jvm.options -在此文件中设置 JVM 选项。

  3. bootstrap.properties -此文件会影响 Open Liberty 服务器的启动。

  4. server.xml -此必需文件指定服务器配置和功能部件。

server.env

server.env 文件是可选的。 这些文件由 bin/server shell 脚本读取，并指定主要用于影响 bin/server 脚本行为的环境变量。 将按顺序从以下位置读取 server.env 文件:

  1. ${wlp.install.dir}/etc/

  2. ${wlp.user.dir}/shared/

  3. ${server.config.dir}/

如果在多个位置中设置了同一属性，那么将使用找到的最后一个值。 还可以在 shell 环境中指定这些环境变量，但 server.env 文件优先于这些变量。

这些文件最常见的用途是设置以下环境变量:

  • JAVA_HOME -指示要使用的 JVM。 如果未设置 JVM ，那么将使用系统缺省值。

  • WLP_USER_DIR -指示包含服务器配置的 usr 目录的位置。 仅在 etc/server.env 文件中设置此环境变量，因为其他位置相对于 usr 目录。

  • WLP_OUTPUT_DIR -指示服务器写入文件的位置。 缺省情况下，服务器将写入从中读取配置的目录结构。 但是，在某些安全概要文件中，服务器配置是只读的，因此服务器必须将文件写入其他位置。

server.env 文件采用 KEY=value 格式，如以下示例中所示:

JAVA_HOME=/opt/ibm/java
WLP_USER_DIR=/opt/wlp-usr

以下限制适用于 server.env 文件:

  • 键值不得包含任何空格。

  • 空行将被忽略。

  • 以数字符号 (#) 开头的行 将被忽略，但在 Linux上启用变量扩展的 # enable_variable_expansion 行除外。

  • 确保等号 (=) 周围没有空格。

  • 除非启用了扩展变量，否则所有字符都是文字。

有关更多信息，请参阅 缺省环境变量。

扩展变量

扩展变量是由其值替换的环境或 shell 变量。 可以在其他变量的值定义中指定扩展变量。 这些扩展变量提供的值可以定义为服务器配置中的环境变量或 shell 环境中的 shell 变量。 插入标记符号 (^) 用于在 Windows 上对惊叹号 (!) 之类的变量扩展字符进行转义。 用于指定扩展变量的语法依赖于操作系统。

缺省情况下，在 Windows 上启用了变量扩展支持。 您可以使用 Windows 延迟扩展语法 !variable_name!来指定扩展变量，该语法将替换为该变量的值。

以下 server.env 文件示例显示了 Windows 上扩展变量的语法。

IBM_DIR=!ProgramFiles!\IBM
JAVA_HOME=!IBM_DIR!\java
WLP_USER_DIR=!USERPROFILE!\wlp-usr

在 V 22.0.0.1 和更高版本中， Open Liberty 在 Linux 系统上支持 server.env 文件中的扩展变量。 扩展变量以美元符号 ($) 作为前缀 并且可以选择将变量名称括在花括号 ({}) 中，例如 $variable_name 或 ${variable_name}。

为 Linux启用扩展变量时，请使用单引号 (' ) ，双引号 (" ) 和反斜杠 (\) 行为与 shell 环境中的行为相同。 例如，将字符串放在单引号中会阻止变量的扩展。 将字符串放在双引号中并不会阻止变量的扩展，反斜杠是一个转义字符，会除去特殊字符的含义。

缺省情况下， server.env 文件在 Linux上不支持变量扩展。 每个 KEY=value 对中的值按字面意思解释，您不需要对特殊字符 (例如空格) 进行转义。 但是，当您启用对变量扩展的支持时，必须对特殊字符进行转义。

要启用对变量扩展的支持，请在 server.env 文件开头添加以下注释。

# enable_variable_expansion

当启用扩展变量时，必须将包含空格的任何值括在引号中。 您还必须对内部引号进行转义。 例如，内部引号在以下值中转义:

MESSAGE1='Don\'t do that'

以下 server.env 文件示例显示了 Linux上扩展变量的语法。 WLP_USER_DIR 和 LOG_DIR 环境变量的值包含扩展变量。

# enable_variable_expansion
JAVA_HOME=/opt/ibm/java
WLP_USER_DIR=/home/${USER}/wlp-usr
LOG_DIR=${WLP_USER_DIR}/logs

jvm.options

jvm.options 文件是可选的。 这些文件由 bin/server shell 脚本读取，以确定针对 Open Liberty启动 JVM 时要使用的选项。 按顺序从以下位置读取 jvm.options 文件:

  1. ${wlp.user.dir}/shared/jvm.options

  2. ${server.config.dir}/configDropins/defaults/

  3. ${server.config.dir}/

  4. ${server.config.dir}/configDropins/overrides/

如果这些位置中不存在 jvm.options 文件，那么服务器脚本会在 ${wlp.install.dir}/etc中查找该文件 (如果存在此类目录)。

jvm.options 文件的常见用途包括:

  • 设置 JVM 内存限制

  • 启用由监视产品提供的 Java 代理程序

  • 设置 Java 系统属性

jvm.options 文件格式对每个 JVM 选项使用一行，如以下示例中所示:

-Xmx512m
-Dmy.system.prop=This is the value.

您不需要对特殊字符 (例如空格) 进行转义。 将按顺序读取选项并将其提供给 JVM。 如果提供多个选项，那么 JVM 会看到所有选项。 这些文件不支持变量替换。

bootstrap.properties

bootstrap.properties 文件是可选的。

此文件在 Open Liberty 引导期间读取，以提供服务器启动的最早阶段的配置。 它由早于 server.xml 文件的服务器读取，因此可以从一开始就影响 Open Liberty 内核的启动和行为。 bootstrap.properties 文件是一个简单的 Java 属性文件，位于 ${server.config.dir}中。 bootstrap.properties 文件的常见用途是配置日志记录，因为它可能会在读取 server.xml 文件之前影响日志记录行为。

bootstrap.properties 文件支持特殊可选属性 bootstrap.include，该属性指定在引导阶段也要读取的另一个属性文件。 例如，此 boostrap.include 文件可以包含一组公共引导属性，供多个服务器使用。 将 bootstrap.include 文件设置为绝对或相对文件路径。

server.xml

最重要且唯一需要的配置文件是 server.xml 文件。 server.xml 文件必须是格式正确的 XML ，并且根元素必须是 server。 服务器支持的确切元素取决于配置的功能部件，并且将忽略任何未知配置。

Open Liberty 使用异常配置原则，这允许使用简洁的配置文件。 运行时环境通过一组内置配置缺省设置运行。 仅指定用于覆盖这些缺省设置的配置。

按顺序从以下位置读取服务器配置文件:

  • ${server.config.dir}/configDropins/defaults/

  • ${server.config.dir}/server.xml

  • ${server.config.dir}/configDropins/overrides/

${server.config.dir}/server.xml 文件必须存在，但其他文件是可选的。

您可以通过将服务器格式的 XML 文件删除到目录中来灵活编写配置。 将在两个 configDropins 目录中按字母顺序读取文件。

configDropins 目录提供了一种方便的方法来添加或修改服务器配置，而无需直接更改 server.xml 文件。 它位于 usr/servers/server_name下，包含两种类型的子目录:

  • 覆盖 (usr/servers/server_name/configDropins/overrides): 此目录用于覆盖 server.xml 文件中的配置。 例如，要更改 server.xml中定义的端口配置，请将相关配置文件放在此处。

  • 缺省值 (usr/servers/server_name/configDropins/defaults): 此目录用于设置 server.xml中未指定的配置元素的缺省值。 它允许 server.xml 保留主配置源，并通过此目录提供其他缺省设置。

将动态监视对放置在 configDropins/overrides 或 configDropins/defaults 目录中的配置文件所作的更改。 添加，移除或更新会自动反映在运行时配置中。

configDropins/overrides 目录中的配置优先于 server.xml 文件中的配置，而后者又优先于 configDropins/defaults 目录中的配置。 configDropins/overrides 和 configDropins/defaults 中的配置将取代功能部件指定的任何缺省设置。

为了保持不同平台之间的一致性，首先将文件名转换为小写，然后排序。 在两个文件共享相同名称但大小写不同 (例如 extraConfig.xml 和 ExtraConfig.xml) 的场景中，它们的处理顺序可能不一致。

配置变量语法

变量定义在运行时用实际值替换的可复用值。 在 Open Liberty 服务器配置文件中，可以使用 ${variable.name} 语法来引用变量。 美元符号 ($) 指示变量引用的开始。 大括号 ({}) 将变量的名称括起来。

例如，如果定义名为 database.url 且值为 jdbc:mysql://localhost:3306/mydb的变量，那么可以使用 ${database.url}在服务器配置文件中引用该变量。 在运行时， Open Liberty 将 ${database.url} 替换为实际值 jdbc:mysql://localhost:3306/mydb。

有关指定变量的更多信息，请参阅 变量替换优先顺序。

变量替换优先顺序

您可以使用变量来参数化服务器配置。 要解析对其值的变量引用，请按顺序查阅下列源，按优先顺序递增，这意味着以后的源将取代先前的源并优先于先前的源:

  1. server.xml 文件中的变量缺省值

  2. 环境变量

  3. bootstrap.properties

  4. Java 系统属性

  5. 从 ${server.config.dir}/variables 目录或其他目录中的文件装入的变量 (由 VARIABLE_SOURCE_DIRS 环境变量指定)

  6. server.xml 文件中声明的变量值

  7. 在命令行上声明的变量

通过使用 ${variableName} 语法来引用变量。 在服务器配置中指定变量，如以下示例中所示:

<variable name="variableName" value="some.value" />

仅当未指定其他值时，才会使用 server.xml 文件中指定的缺省值:

<variable name="variableName" defaultValue="some.default.value" />

您还可以从命令行在启动时指定变量。 如果执行此操作，那么在命令行上指定的变量将覆盖所有其他变量源，并且在服务器启动后无法更改:

server run myserver -- --variableName=variableValue

可以将环境变量作为变量进行访问。 您可以直接引用环境变量名称。 如果无法按指定对变量进行解析，那么 server.xml 文件将在环境变量名称上查找以下变体:

  • 将所有非字母数字字符替换为下划线字符 (_)

  • 将所有字符更改为大写

例如，如果在 server.xml 文件中输入 ${my.env.var} ，那么它将查找具有以下名称的环境变量:

  1. my.env.var

  2. my_env_var

  3. MY_ENV_VAR

VARIABLE_SOURCE_DIRS 环境变量指定可以从中装入变量的目录。 如果未指定此环境变量，那么变量目录的缺省位置为 ${server.config.dir}/variables。 要将目录列表定义为此变量的值，请使用操作系统的路径分隔符来分隔每个目录。 对于 Windows ，路径分隔符为分号 (;)。 对于基于 Unix 的系统，路径分隔符为冒号 (:)。

文件的名称用作变量名，文件的内容用作变量值。 例如，您可以在变量目录中创建一个名为 httpPort 的文件，将文本字符串 9080 作为唯一内容。 然后，可以在 server.xml 文件中使用变量 ${httpPort} 并将其解析为 9080。

如果在 variables 目录中的目录中创建文件，那么会将父目录添加到变量名称。 例如，您可以在变量目录中的 ports 子目录中创建 ${httpPort} 文件，变量名称为 ${ports/httpPort}。

还可以在属性文件中定义 VARIABLE_SOURCE_DIRS 目录中的变量。 如果文件名以 .properties结尾，那么将使用文件中的每个属性来创建变量。 例如，可以创建具有以下内容的名为 ports.properties 的文件:

httpPort=9080
httpsPort=9443

通过此配置，您可以在 server.xml 文件中指定 ${httpPort} 和 ${httpsPort} 变量，并将它们的值解析为 9080 和 9443。

变量值始终解释为具有简单类型转换的字符串。 因此，可以将端口列表 (例如 80,443) 解释为单个字符串，而不是两个端口号。 您可以使用 list 函数在 , 上强制拆分变量替换，如以下示例中所示:

<mongo ports="${list(mongoPorts)}" hosts="${list(mongoHosts)}" />

具有整数值的变量支持简单算术。 运算符的左侧和右侧可以是变量或数字。 运算符可以是 +， -， *或 /，如以下示例中所示:

<variable name="one" value="1" />
<variable name="two" value="${one+1}" />
<variable name="three" value="${one+two}" />
<variable name="six" value="${two*three}" />
<variable name="five" value="${six-one}" />
<variable name="threeagain" value="${six/two}" />

存在以下预定义变量:

  • wlp.install.dir -安装 Open Liberty 运行时的目录。

  • wlp.server.name -服务器的名称。

  • wlp.user.dir - usr 文件夹的目录。 缺省值为 ${wlp.install.dir}/usr。

  • shared.app.dir -共享应用程序的目录。 缺省值为 ${wlp.user.dir}/shared/apps。

  • shared.config.dir -共享配置文件的目录。 缺省值为 ${wlp.user.dir}/shared/config。

  • shared.resource.dir -共享资源文件的目录。 缺省值为 ${wlp.user.dir}/shared/resources。

  • server.config.dir -存储服务器配置的目录。 缺省值为 ${wlp.user.dir}/servers/${wlp.server.name}。

  • server.output.dir -服务器用于写入工作区，日志和其他运行时生成的文件的目录。 缺省值为 ${server.config.dir}。

配置合并

由于配置可以由多个文件组成，因此两个文件可能提供相同的配置。 在这些情况下，将根据一组简单规则来合并服务器配置。 在 Open Liberty中，配置分为单项配置和工厂配置，每个配置都有自己的合并规则。 单项配置用于配置单个元素 (例如，日志记录)。 工厂配置用于配置多个实体，例如整个应用程序或数据源。

合并单项配置

对于多次指定的单项配置元素，将合并配置。 如果存在两个具有不同属性的元素，那么将使用这两个属性。 例如：

<server>
    <logging a="true" />
    <logging b="false" />
</server>

被视为:

<server>
    <logging a="true" b="false" />
</server>

如果两次指定同一属性，那么最后一个实例优先。 例如：

<server>
    <logging a="true" b="true" />
    <logging b="false" />
</server>

被视为:

<server>
    <logging a="true" b="false" />
</server>

有时通过使用获取文本的子元素来提供配置。

在这些情况下，将使用指定的所有值来合并配置。 最常见的方案是配置功能部件。 例如：

<server>
    <featureManager>
        <feature>servlet-4.0</feature>
    </featureManager>
    <featureManager>
        <feature>restConnector-2.0</feature>
    </featureManager>
</server>

被视为:

<server>
    <featureManager>
        <feature>servlet-4.0</feature>
        <feature>restConnector-2.0</feature>
    </featureManager>
</server>

正在合并工厂配置

工厂配置合并使用与单项配置相同的规则，但不会仅因为元素名称匹配而自动合并元素。 使用工厂配置时，配置同一元素是有效的，并且意味着两个不同的逻辑对象。 因此，假定每个元素都配置一个不同的对象。 如果单个逻辑对象由两个元素配置，那么必须在每个元素上设置 id 属性以指示它们是相同的。 不支持 id 属性上的变量替换。

以下示例配置两个应用程序。 第一个应用程序是 myapp.war，它的上下文根为 myawesomeapp。 另一个应用程序是 myapp2.war，它将 myapp2 作为上下文根:

<server>
    <webApplication id="app1" location="myapp.war" />
    <webApplication location="myapp2.war" />
    <webApplication id="app1" contextRoot="/myawesomeapp" />
</server>

包含处理

您可以使用 include 元素在服务器配置中包含其他配置资源。 当服务器配置文件包含对另一个资源的包含引用时，服务器将处理所引用资源的内容，就像它们以内联方式包含在 include 元素中一样。

在以下示例中，服务器先处理 other.xml 文件的内容，然后再处理 other2.xml 文件的内容:

<server>
    <include location="other.xml" />
    <include location="other2.xml" />
</server>

可以将 include 元素的 location 属性设置为相对或绝对文件路径，目录或远程资源的 URI。

将目录指定为 location 值时，服务器将仅处理直接位于指定目录中的文件，并忽略任何子目录。 服务器按字母顺序处理目录中的文件。

<server>
    <include location="./common/"/>
</server>

在此示例中，将在服务器启动时按字母顺序处理 ./common/ 目录中的配置文件。

当服务器处理目录中的配置文件时，它会将文件名切换为小写，并按字母顺序对其进行排序，以确保文件处理顺序一致且可预测。 但是，如果在同一目录中指定了两个同名文件，但大小写变体 (例如 extraConfig.xml 和 ExtraConfig.xml) 除外，那么排序行为是不确定的。

在 include 位置中指定目录时，必须向目录名称追加斜杠 (/)。 例如，location="./common/"。 在目录名 (location="./common") 后省略斜杠会导致错误。

包含来自远程资源的 URI 的配置时， location 属性必须指向单个远程资源文件而不是目录。 例如，支持 https://website.com/path/to/server.xml ，但不支持 https://website.com/path/to/directory 。

缺省情况下，包含资源必须存在。 如果资源可能不可用，请将 optional 属性设置为 true，如以下示例中所示:

<server>
    <include location="other.xml" optional="true" />
</server>

包含资源时，可以指定 onConflict 属性以更改常规合并规则。 您可以将 onConflict 属性的值设置为 IGNORE 或 REPLACE 任何冲突的配置:

<server>
    <include location="other.xml" onConflict="IGNORE" />
    <include location="other2.xml" onConflict="REPLACE" />
</server>

对于使用 include 元素从目录引入的配置文件， onConflict 属性将单独应用于该目录中的每个文件。 例如：

<server>
    <include location="/directory/" onConflict="REPLACE" />
</server>

在功能上，此配置与使用其自己的 onConflict 属性单独包含每个文件相同，如以下示例中所示:

<server>
    <include location="a.xml" onConflict="REPLACE" />
    <include location="b.xml" onConflict="REPLACE" />
</server>

配置引用

Open Liberty 中的大多数配置都是自包含的，但共享配置通常很有用。 例如， JDBC 驱动程序配置可能由多个数据源共享。 您可以引用定义为 server 元素的直接子代的任何工厂配置元素。

对配置的引用始终使用所引用元素的 id 属性。 使引用的配置元素使用始终以 Ref结尾的属性，如以下示例中所示:

<server>
  <dataSource jndiName="jdbc/fred" jdbcDriverRef="myDriver" />
  <jdbcDriver id="myDriver" />
</server>

动态更新

服务器监视服务器 XML 配置以获取更新，并在检测到更改时动态重新装入。 对非 XML 文件 (server.env， bootstrap.properties和 jvm.options) 的更改不是动态的，因为它们仅在启动时读取。 将每隔 500ms监视本地磁盘上的任何服务器 XML 配置文件的更新。 您可以配置 XML 配置文件监视的频率。 例如，要将服务器配置为每 10 分钟监视一次，请指定:

<config monitorInterval="10m" />

要禁用文件系统轮询并仅在通知 MBean 时重新装入，请指定:

<config updateTrigger="mbean" />

日志消息

当服务器运行时，它可能会输出引用配置的日志消息。 日志中的引用使用类似于 XPath 的结构来指定配置元素。 元素名称由方括号内的 id 属性值提供。 如果未在服务器配置中指定 id ，那么将自动生成 id 。 根据以下服务器 XML 配置示例， dataStore 元素和子代 dataSource 在日志中标识为 dataStore[myDS] 和 dataStore[myDS]/dataSource[default-0]。

<server>
    <dataStore id="myDS">
        <dataSource />
    </dataStore>
</server>
