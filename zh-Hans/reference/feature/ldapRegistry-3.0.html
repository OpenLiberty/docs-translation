LDAP 用户注册表 3.0

此功能部件启用将 LDAP 服务器用作用户注册表的支持。 可以使用任何支持 LDAP V3.0 的服务器。 可以配置多个 LDAP 用户注册表，然后对它们进行联合以获得单个逻辑用户注册表视图。

启用此功能

要启用 LDAP 用户注册表 3.0 功能，请将以下元素声明添加到 server.xml 文件中 featureManager 元素内:

<feature>ldapRegistry-3.0</feature>

配置 LDAP 用户注册表

要配置 LDAP 用户注册表，必须指定基本专有名称，主机， LDAP 类型，端口号和域属性。 大多数 LDAP 服务器都需要绑定凭证以向服务器认证客户机。

要使用简单认证进行绑定，请在 ldapRegistry 元素中指定绑定专有名称和密码。 以下示例显示了将 IBM Tivoli Directory Server 用作 LDAP 用户注册表的简单配置。

<ldapRegistry baseDN="o=acme.com" host="ldap.acme.com"
ldapType="IBM Tivoli Directory Server" port="389" realm="AcmeLdap"
bindDN="cn=testuser,o=acme.com" bindPassword="mypassword"/>

将 LDAP 注册表用于管理员角色映射。 将 ldapRegistry-3.0 功能部件添加到 "server.xml" 文件。 以下示例显示了向用户 "bob" 授予管理员角色的 LDAP 注册表。

<ldapRegistry id="basic" host="" port="">
    <tds.properties ... />
</ldapRegistry>

<administrator-role>
	<user>cn=bob,o=ibm,c=us</user>
</administrator-role>

您可以使用 securityUtility 编码 命令对 bindPassword 密码进行编码。 运行 securityUtility encode 命令时，请提供要作为命令行输入进行编码的密码，如果未指定任何参数，那么该命令将提示您输入密码。 然后，该命令将输出已编码的值。 复制命令输出的此编码值，并将此值用于 bindPassword 密码。

有关对 LDAP 进行故障诊断的更多信息，请参阅 对 LDAP 进行故障诊断。

定制搜索过滤器

您可以定制组和用户过滤器，以在 LDAP 用户注册表的搜索中包含定制类或属性。 用于指定过滤器的配置因您指定的 LDAP 类型而异。 每种 LDAP 类型都具有一组不同的过滤器属性。 有关 LDAP 类型的更多信息，请参阅 功能部件配置元素。 将 %v 变量用于要在运行时替换为特定登录名或组名的属性值断言。 以下示例显示为 IBM Tivoli Directory Server LDAP 类型指定的定制过滤器和组过滤器。

<ldapRegistry baseDN="o=acme.com" host="ldap.acme.com"
		ldapType="IBM Tivoli Directory Server" port="389" realm="AcmeLdap"
		bindDN="cn=testuser,o=acme.com" bindPassword="mypassword">
    <idsFilters
		    groupFilter="(&amp;(cn=%v)(objectclass=groupofnames))"
		    userFilter="(&amp;(objectclass=inetorgperson)(|(uid=%v)(mail=%v)))" />
</ldapRegistry>

使用 LDAPS 配置安全连接

要启用与 LDAP 服务器的安全连接以使用 TLS 或 SSL 协议进行加密通信，请配置 ldapRegistry 元素以启用轻量级目录访问协议 over Secure Socket Links (LDAPS)。 将 sslEnabled 属性设置为 true ，将 sslRef 属性设置为 ssl id 值。 要启用安全 TLS 或 SSL 连接，必须在 ssl 元素中配置这些连接。 有关 TLS 和 SSL 配置的更多信息，请参阅 SSL 指令表。

以下示例显示了 ldapRegistry 元素和 LDAP 连接所引用的 ssl 元素。 在 ssl 元素中， id 参数标识配置，以便 ldapRegistry 元素的 sslRef 参数可以引用该配置。 keyStoreRef 和 trustStoreRef 参数建立 TLS 或 SSL 握手中使用的数据源，这将启用安全连接。

<ldapRegistry host="ldapserver.mycity.mycompany.com" port="389"
                         sslEnabled="true" sslRef="LDAPSSLSettings"..../ >

<ssl id="LDAPSSLSettings" keyStoreRef="LDAPKeyStore" trustStoreRef="LDAPTrustStore" />
<keyStore id="LDAPKeyStore" location="server1/resources/security/key.p12" type="PKCS12" password="password" />
<keyStore id="LDAPTrustStore" location="server1/resources/security/trust.p12" type="PKCS12" password="password" />

或者，可以启用用于连接到 LDAP 服务器的 SSL 配置，而不指定 sslRef 属性。 当 sslEnabled 属性设置为 true 并且未指定 sslRef 属性时， SSL 代码将查找与 LDAP 主机和端口匹配的出站过滤器。 如果不存在匹配的过滤器，那么将使用 SSL 出站缺省值。

在以下示例中，已启用 SSL ，但未指定 sslRef 属性，因此 LDAP 使用 LdapSSLSettings SSL 配置，该配置指定与 ldapRegistry 配置相同的主机和端口。

<ldapRegistry host="ldapserver.mycity.mycompany.com" port="389" sslEnabled="true"... / >

<ssl id="LdapSSLSettings" keyStoreRef="LDAPKeyStore" trustStoreRef="LDAPTrustStore" >
    <outboundConnection host="ldapserver.mycity.mycompany.com" port="389" />
</ssl>
<keyStore id="LDAPKeyStore" location="server1/resources/security/key.p12" type="PKCS12" password="password" />
<keyStore id="LDAPTrustStore" location="server1/resources/security/trust.p12" type="PKCS12" password="password" />

指定定制 LDAP 对象类

您可以指定 LDAP 实体类型以配置对定制或非缺省对象类的支持。 实体类型将不同的对象类分组为单个实体，以便服务器在同一实体名称下识别这些对象类。 您还可以定制任何实体类型的搜索条件，以缩短其包含的类的搜索时间。 如果为用户或组创建定制对象类，请更新组和用户过滤器以包含新类。 使用 LDAP 实体来指定多个搜索条件，而不是为同一 LDAP 服务器创建具有不同 baseDNs 的多个 LDAP 注册表。

以下示例显示了具有定制搜索条件的用户和组的实体类型。

<ldapRegistry baseDN="o=acme.com" host="ldap.acme.com"
		ldapType="IBM Tivoli Directory Server" port="389" realm="AcmeLdap"
		bindDN="cn=testuser,o=acme.com" bindPassword="mypassword">
		<ldapEntityType name="PersonAccount">
			<objectClass>inetorgperson</objectClass>
			<objectClass>customuser</objectClass>
			<searchBase>ou=users,o=acme.com,c=us</searchBase>
		</ldapEntityType>
		<ldapEntityType name="Group">
			<objectClass>groupofnames</objectClass>
			<objectClass>customgroup</objectClass>
			<searchBase>ou=groups,o=acme.com,c=us</searchBase>
		</ldapEntityType>
		<idsFilters
			groupFilter="(&amp;(cn=%v)(|(objectclass=groupofnames)(objectclass=customgroup)))"
			userFilter="(&amp;(|(objectclass=inetorgperson)(objectclass=customuser))(|(uid=%v)(mail=%v)))" />
</ldapRegistry>

用户和组过滤器包含定制对象类。 虽然此示例显示了 IBM Tivoli Directory Server LDAP 类型的配置，但所有 LDAP 类型都支持 userFilter 和 groupFilter 属性。

在上一个示例中， ObjectClass 的值包括 inetorgperson 或 groupofnames。 如果未定义正确的 ObjectClass 或 ObjectCategory 值，那么会发生 ClassCastException ，如以下示例中所示。

java.lang.ClassCastException: com.ibm.wsspi.security.wim.model.Entity is not compatible with com.ibm.wsspi.security.wim.model.LoginAccount.

在未正确定义 ObjectClass 的其他情况下，可能找不到用户，或者对 getSecurityName 的请求可能返回空字符串。

searchBase 为给定实体类型的搜索调用指定 LDAP 服务器的子树，这将覆盖搜索操作中的 baseDN 。 例如，如果 baseDN 为 o=acme.com，c=us 并且 PersonAccount 实体类型的搜索条件定义为 ou=users，o=acme.com和c=us，那么将在 ou=users，o=ibm和c=us 子树下进行 PersonAccount 的所有搜索调用。 将忽略对其他子树 (例如 ou=iThings，o=ibm和c=us) 的调用。

联合 LDAP 用户注册表

缺省情况下，将联合 LDAP 用户注册表。 如果在 server.xml 文件中配置多个 LDAP 用户注册表，那么用户注册表将自动联合到公共用户注册表中。 请确保这些用户在所有联合存储库中唯一，否则用户注册表操作不会成功。 使用多个联合 LDAP 存储库时，每个存储库都必须定义唯一的 baseDN 属性。

通过指定 federatedRepository 元素，可以在启用 联合用户注册表 功能部件时调整联合注册表的配置。 如果启用 LDAP 用户注册表功能部件 V 3.0 或更高版本，那么缺省情况下将启用联合用户注册表功能部件。 否则，必须手动启用 "联合用户注册表" 功能以使用 federatedRepository 元素。

如果指定了 federatedRepository 元素以配置 participatingBaseEntry 和 primaryRealm 元素，那么将仅对 primaryRealm 元素中定义的存储库执行用户注册表操作。 可在 primaryRealm 元素下对不同用户注册表 API 定义输入和输出属性映射。

以下示例显示了使用 federatedRepository 元素中指定的配置自动联合的两个 LDAP 注册表。

<ldapRegistry host="ldapserver1.mycity1.mycompany.com" baseDN="o=mycompany,ou=myou,c=us"
    port="123" ldapType="IBM Tivoli Directory Server" name="o=mybaseentry">
</ldapRegistry>

<ldapRegistry host="ldapserver2.mycity2.mycompany.com"
    baseDN="cn=users,dc=secfvt2,dc=mycity2,dc=mycompany,dc=com"
    port="456"
    ldapType="Microsoft Active Directory"
    bindDN="cn=testuser,cn=users,dc=secfvt2,dc=mycity2,dc=mycompany,dc=com"
    bindPassword="{xor}KzosKyosOi0vKDs=">
</ldapRegistry>

<federatedRepository>
	 <primaryRealm name="RealmName" delimiter="@" allowOpIfRepoDown="true">
	 	<participatingBaseEntry name="o=mybaseentry"/>
		 <participatingBaseEntry name="cn=users,dc=secfvt2,dc=mycity2,dc=mycompany,dc=com"/>
	 	 <uniqueUserIdMapping inputProperty="uniqueName" outputProperty="uniqueName"/>
	 	 <userSecurityNameMapping inputProperty="principalName" outputProperty="principalName"/>
        		 <userDisplayNameMapping inputProperty="principalName" outputProperty="principalName"/>
		 <uniqueGroupIdMapping inputProperty="uniqueName" outputProperty="uniqueName"/>
        		 <groupSecurityNameMapping inputProperty="cn" outputProperty="cn"/>
        		 <groupDisplayNameMapping inputProperty="cn" outputProperty="cn"/>
        	</primaryRealm>
</federatedRepository>

ldapRegistry 元素的 name 属性是可选的。 如果指定了此属性，那么 participatingBaseEntry 元素中 name 属性的值必须与 ldapRegistry 元素中 name 属性的值匹配。 如果未指定 ldapRegistry 元素中的 name 属性，那么 participatingBaseEntry 元素中 name 属性的值必须与 ldapRegistry 元素中 baseDN 属性的值匹配。

在上一个示例中的其中一个 participatingBaseEntry 元素配置上演示了这些选项中的每个选项。

您还可以将 LDAP 用户注册表与基本或定制用户注册表联合。 用户注册表的参与基本条目由 participatingBaseEntry 元素定义。 定制或基本用户注册表的参与基本条目值是设置为等于该用户注册表的域名的 o 组织属性。 对于 LDAP 用户注册表，域名是 LDAP 用户注册表配置中的基本专有名称。 要验证用户在公共用户注册表中是否唯一，每个搜索请求都会搜索所有联合用户注册表。 缺省情况下，所有联合用户注册表都必须成功返回，否则请求将失败。

为 LDAP 服务器配置 Kerberos 认证

要为 LDAP 服务器配置 Kerberos 绑定认证，必须在 ldapRegistry 元素上配置绑定认证机制和 Kerberos 主体，如以下示例中所示:

<ldapRegistry id="LDAP" realm="SampleLdapADRealm" host="ldap_hostname" port="389"
	ignoreCase="true"  baseDN="DC=example,DC=com" bindAuthMechanism="GSSAPI"
	krb5Principal="user1@EXAMPLE.COM" krb5TicketCache="${server.config.dir}/security/krb5-user1.cc"
	ldapType="Custom" />

Kerberos 主体由必需的 krb5Principal 属性指定。 必须将 bindAuthMechanism 属性设置为 GSSAPI 值。 此绑定认证机制是 简单绑定认证机制的替代方法，它使用绑定专有名称和绑定密码。

krb5TicketCache 属性是可选属性，用于指定 ccache 文件的位置，该文件是凭证高速缓存文件。 ccache 文件中的凭证可能会到期。 当指定了 krb5TicketCache 属性并对主体进行了认证时， Kerberos 服务会自动尝试在凭证到期之前对其进行更新。

或者，可以在 server.xml 文件中指定 kerberos 元素，以便为所有使用 Kerberos 凭证的功能部件配置 Kerberos 认证。 此元素用于配置 keytab 文件和配置文件，这些文件可以向使用 Kerberos 凭证的任何 Open Liberty 功能部件提供值。 kerberos 元素是可选的。 有关更多信息，请参阅 Open Liberty的Kerberos 认证。

如果未指定 krb5TicketCache 属性，那么 Open Liberty 将解析 kerberos 元素中配置的 Kerberos keytab 文件中的凭证值。 如果未配置 keytab 文件或 krb5TicketCache 属性，那么 Open Liberty 将从 Java SDK 缺省设置指定的凭证高速缓存位置解析凭证值。 如果同时配置了 kerberos 元素中的 krb5TicketCache 属性和 keytab 属性，那么将在这两个文件中搜索凭证。 Open Liberty 首先在 krb5TicketCache 属性定义的 ccache 文件中搜索，然后在 kerberos 元素定义的 keytab 文件中搜索。

可以选择为使用 Kerberos 凭证的任何功能部件指定 krb5TicketCache 属性。 如果指定了此属性，那么它优先于任何 keytab 和 configFile 值， Java SDK 缺省值或操作系统缺省值。 您可以指定此属性以配置与 kerberos 元素中配置的值不同的特定功能部件的凭证。

要确定与 LDAP 服务器的 Kerberos 认证相关联的常见问题和错误消息的原因，请参阅 对 LDAP 服务器的 Kerberos 认证进行故障诊断。

定义客户机证书属性与 LDAP 用户注册表属性之间的映射

要将客户机 X.509 证书中的属性映射到 LDAP 配置中的属性，可以指定 CERTIFICATE_FILTER 映射方式。

如果在运行时有多个 LDAP 条目与过滤器指定项匹配，那么认证就会失败，因为这会导致模糊匹配。 此过滤器的语法是： LDAP attribute=${Client certificate attribute}

uid=${SubjectCN} 是简单证书过滤器的一个示例。

另外，还可以在证书过滤器中指定多个属性和值。 过滤器指定项的 LDAP 属性取决于 LDAP 服务器依据配置所要使用的模式。 客户机证书属性是客户机证书中的一个公共属性。 客户机证书属性必须以美元符号开头，并且必须括在花括号中，例如 ${SubjectCN}。 这些属性区分大小写。

支持的 LDAP 属性包括 uid， initials， sAMAccountName， displayName， distinguishedName， displayName和 description。

受支持的客户机证书属性为 ${SubjectCN}， ${SubjectDN}， ${IssuerCN}， ${IssuerDN}和 ${SerialNumber}。

以下示例显示具有 certificateMapMode 属性启用的证书过滤方式的 LDAP 配置以及由 certificateFilter 属性指定的证书过滤器。 在此证书过滤器配置中，这是 uid LDAP 属性到 ${SubjectCN} 客户机证书属性的值。

<ldapRegistry id="LDAP" realm="SampleLdapIDSRealm"
      host="myldap.ibm.com" port="389" ignoreCase="true"
      baseDN="o=ibm,c=us"
      ldapType="IBM Tivoli Directory Server" searchTimeout="8m"
      certificateMapMode="CERTIFICATE_FILTER"
      certificateFilter="uid=${SubjectCN}">
      <idsFilters
      userFilter="(&amp;(uid=%v)(objectclass=ePerson))"
      groupFilter="(&amp;(cn=%v)(|(objectclass=groupOfNames)
          (objectclass=groupOfUniqueNames)(objectclass=groupOfURLs)))"
      userIdMap="*:uid"
      groupIdMap="*:cn"
      groupMemberIdMap="ibm-allGroups:member;ibm-allGroups:uniqueMember;
          groupOfNames:member;groupOfUniqueNames:uniqueMember">
</idsFilters>
      </ldapRegistry>

有关更多信息，请参阅 LDAP 用户注册表 中的 certificateFilter 和 certificateMapMode

定义 LDAP 属性与用户注册表模式属性之间的映射

您可以定义 LDAP 属性与用户注册表属性之间的映射。 配置映射后，当您将用户注册表属性用于任何操作时，该值等同于映射的 LDAP 属性的值。

在以下示例中，使用 server.xml 文件中的 password 用户注册表属性为 userPassword LDAP 属性定义了映射。 defaultValue 属性是可选的。 为具有 PersonAccount 实体类型的 distinguishedName LDAP 属性的 externalId 用户注册表属性定义了映射。

<ldapRegistry id="LDAP" realm="SampleLdapIDSRealm"
	host="myldap.ibm.com" port="389" ignoreCase="true"
	baseDN="o=ibm,c=us"
	ldapType="IBM Tivoli Directory Server" searchTimeout="8m">
	<attributeConfiguration>
		<attribute name="userPassword" propertyName="password" entityType="PersonAccount" defaultValue="xyz123"/>
		<externalIdAttribute name="distinguishedName" entityType="PersonAccount"/>
	</attributeConfiguration>
</ldapRegistry>

为多个 LDAP 服务器配置故障转移

您可以指定 LDAP 故障转移服务器的配置属性。 这些服务器被指定为准备在主 LDAP 服务器脱机时自动切换并无缝接管的备份服务器。 以下示例显示了主 LDAP 服务器和 ldapRegistry 元素中指定的两组 LDAP failoverServers 元素。 这些 failoverServers 元素具有在其中定义的多个 server 元素。 这些 server 元素充当备份服务器，以防主 LDAP 服务器脱机。

<ldapRegistry id="LDAP" realm="SampleLdapIDSRealm"
    	host="ldapserver1.mycity.mycompany.com" port="389" ignoreCase="true"
     	baseDN="o=ibm,c=us" ldapType="IBM Tivoli Directory Server" idsFilters="ibm_dir_server">
	<failoverServers name="failoverLdapServersGroup1">
		<server host="ldapserver2.mycity.mycompany.com" port="389" />
		<server host="ldapserver3.mycity.mycompany.com" port="389" />
	</failoverServers>
	<failoverServers name="failoverLdapServersGroup2">
		<server host="ldapserver4.mycity.mycompany.com" port="389" />
	</failoverServers>
</ldapRegistry>

<idsLdapFilterProperties id="ibm_dir_server"
	    userFilter="(&amp;(uid=%v)(objectclass=ePerson))"
	    groupFilter="(&amp;(cn=%v)(|(objectclass=groupOfNames)
                 (objectclass=groupOfUniqueNames)(objectclass=groupOfURLs)))"
	    userIdMap="*:uid" groupIdMap="*:cn"
	    groupMemberIdMap="ibm-allGroups:member;ibm-allGroups:uniqueMember;
                      groupOfNames:member;groupOfUniqueNames:uniqueMember">
</idsLdapFilterProperties>
