OpenID Connect 客户机
1.0

此功能部件允许 Web 应用程序集成 OpenID Connect 客户机 1.0 以认证用户而不是（或以及）所配置用户注册表。

您可以将 Open Liberty 服务器配置为充当 OpenID Connect 客户机或依赖方 (RP) ，以便可以将 OpenID Connect 提供者 (OP) 用作单点登录身份提供者。

您可以在 Web 或 RESTful Web Services (JAX-RS) 应用程序中使用 OpenIdAuthenticationMechanismDefinition 注释来为该应用程序启用 OpenID Connect 客户机。 当您启用 Application Security 功能部件 V 5.0 或更高版本时，支持此注释。 您无需启用 OpenID Connect Client 功能部件即可使用此注释。 有关更多信息，请参阅 为应用程序启用 OpenID Connect 客户机。

启用此功能

要启用 OpenID 连接客户端 1.0 功能，请在 server.xml 文件中的 featureManager 元素内添加以下元素声明：

<feature>openidConnectClient-1.0</feature>

以下示例演示 OpenID Connect 客户机功能部件的配置选项，该功能部件在 server.xml 文件中的 openidConnectClient 元素上指定。 有关配置属性的完整列表，请参阅 openidConnectClient 元素。

  • Open Liberty 的最低配置，以充当 OpenID Connect 客户机

  • 为 OpenID 连接提供程序配置一个发现端点

  • 为 OpenID 连接配置 JSON Web 令牌 (JWT) 身份验证

  • 接受 JWE 格式的令牌

  • 配置第三方 OpenID Connect 提供者

  • 支持多个 OpenID 连接提供程序

  • 使用专用密钥 JWT 进行客户机认证

  • 禁用 LTPA cookie

  • 接受 OAuth 2.0 不记名访问令牌，而不重定向到 OpenID Connect 提供者

  • 修改上下文根目录

  • 将请求参数传递到 OpenID Connect

  • 检查访问令牌以获取用户和组信息

Open Liberty 的最低配置，用于充当 OpenID Connect 客户机

要使 Open Liberty 服务器能够充当 OpenID Connect (OIDC) 客户机，必须配置以下先决条件。

  • 启用 OpenID Connect 客户机和 传输安全性 功能部件。

  • 配置用户注册表。

    默认情况下，OIDC 提供商 (OP) 返回的用户身份不会映射到注册表用户，因此无需在注册表中配置用户。 但是，如果 openidConnectClient 元素的 mapIdentityToRegistryUser 属性设置为 true，那么对于从 OP 返回的身份，必须存在用户条目。

  • 为包含受支持的 OIDC 提供者的签署者证书的服务器配置信任库。 信任库是在 server.xml 文件中的 keyStore 元素上配置的。

    OIDC 使用服务器指定的缺省 TLS 配置。 因此，除非为 OIDC 指定备用 TLS 配置，否则服务器的缺省 TLS 配置必须引用为 OIDC 配置的信任库。 以下示例显示了 server.xml 文件中的缺省 TLS 配置，该配置已修改为包含包含 OIDC 的签署者证书的信任库。

    <sslDefault sslRef="DefaultSSLSettings" />
<ssl id="DefaultSSLSettings" keyStoreRef="myKeyStore" trustStoreRef="OIDCTrustStore" />
<keyStore id="myKeyStore" password="{xor}EzY9Oi0rJg==" type="PKCS12" location="${server.config.dir}/resources/security/BasicKeyStore.p12" />
<keyStore id="OIDCTrustStore" password="{xor}EzY9Oi0rJg==" type="PKCS12" location="${server.config.dir}/resources/security/OIDCTrustStore.p12" />

    如果指定了缺省出站 TLS 配置，那么此配置还必须使用为 OIDC 配置的信任库。 通过从 openidConnectClient 元素的 sslRef 属性引用 OIDC ，可以为 OIDC 指定除缺省配置以外的 TLS 配置。 或者，您可以通过引用 openidConnectClient 元素的 trustStoreRef 属性中的信任库配置来为 OIDC 指定非缺省信任库。 有关详细信息，请参阅 配置 TLS 和 openidConnectClient 元素。

以下示例显示了用于使用缺省 OP 的 openidConnectClient 元素的最低配置。 客户机必须具有在指定 URL 模式中可用的已配置应用程序，该应用程序可以处理来自 OP 的重定向请求。 该 URL 还必须与 OP 为客户端注册的重定向 URL 精确匹配。

<openidConnectClient id="client01"
    clientId="client01"
    clientSecret="{xor}LDo8LTor"
    authorizationEndpointUrl="https://server.example.com:443/oidc/endpoint/OidcConfigSample/authorize"
    tokenEndpointUrl="https://server.example.com:443/oidc/endpoint/OidcConfigSample/token">
</openidConnectClient>

此配置采用以下缺省值。

  • scope=openid profile

    您可以使用 openidConnectClient 元素的 scope 属性来编辑所需的作用域。 例如，您可以将所需范围更改为 openid profile email。 有关 OpenID 作用域的更多信息，请参阅 OpenID Connect 规范。

  • 客户机向 OP 注册 https://<host name>:<ssl port>/oidcclient/redirect/client01 重定向 URL。

    在此 URL 中，将自动解析主机名和 SSL 端口，并且 client01 是 openidConnectClient 元素的 id 属性的值。 如果代理位于 RP 前面，那么可以通过指定 redirectToRPHostAndPort 属性 (例如， redirectToRPHostAndPort="https://<host name>:<ssl port>") 来覆盖主机名和端口。

OpenID Connect RP 无法自动通过代理主机路由请求。 如果使用代理访问 OP，任何与 OP 相关的 URL 属性值都必须包含代理主机和端口，而不是外部 OP 主机和端口。 在大部分情况下，可将 OP 主机和端口替换为代理主机和端口。 您输入的 URL 必须对 RP 以及用于访问由 RP 服务器托管的应用程序的客户机浏览器或应用程序都可视。 有关如何确定要使用的正确 URL 的进一步指南，请联系代理管理员。

下面的示例显示了自动发现 OP 的最小配置。

<openidConnectClient id="RP"
    clientId="ebb996943plm4450bdb674c211a9fab0"
    clientSecret="g12ciZL1zsVYk6yXTMa2iopa8TBwC178CGq42mdJCALYCdQT1kE1ecB010GE"

如果配置了发现端点，那么将忽略任何其他已配置的 OIDC 端点，并将其替换为以下已发现的端点。

  • issuer

  • authorization_endpoint

  • token_endpoint

  • userinfo_endpoint

  • jwks_uri

  • revocation_endpoint

为 OpenID 连接配置 JSON 网络令牌 (JWT) 身份验证

任何拥有 JWT 的受信任方都可以使用该令牌访问 Open Liberty 中的相关资源。 Open Liberty 服务器验证 JWT 并从中创建已认证的主体集。 JWT 必须包含iss、sub和exp声明，并使用公认的签名算法进行签名，才能被接受为身份验证令牌。 公认的签名算法被列为 openidConnectClient 元素的 signatureAlgorithm 属性的可能值。 当 signatureAlgorithm 属性设置为 none时，将接受未签名的 JWT。 还支持 JSON Web 加密 (JWE) 。

要配置 Open Liberty 服务器以接受 JWT 作为认证令牌，必须将 inboundPropagation 属性设置为 required。 您还必须配置包含 JSON Web 密钥 (JWK) 端点证书的信任库，以便 Open Liberty 服务器可以与 JWK 端点建立 TLS 连接。 如果服务器的缺省信任库不包含 JWK 端点证书，那么可以从 openidConnectClient 元素的 trustStoreRef 属性引用备用信任库。

在以下示例中，缺省 TLS 配置包含 JWK 端点证书。 此示例还假定 JWT 签发者支持 JWK ，并且 JWT 使用 RS256 算法进行签名。 issuerIdentifier 属性配置为与 JWT 签发者 iss 声明匹配。

<openidConnectClient id="RS" inboundPropagation="required"
  jwkEndpointUrl="https://example.com/jwtserver/jwk" signatureAlgorithm="RS256"
  issuerIdentifier="https://idp.example.com:8020/jwt" >
</openidConnectClient>

Open Liberty 服务器希望每个请求都能提供有效的 JWT 令牌，并且不会为 JWT 身份验证创建或使用单点登录 (SSO) cookie。 如果希望 Open Liberty 服务器创建 SSO cookie ，请将 openidConnectClient 元素上的 authnSessionDisabled 属性设置为 false。

Open Liberty 可以使用 JWT 或 JSON Web 签名 (JWS) 访问令牌中的签发者声明来选择 server.xml 文件中具有匹配签发者属性的 openidConnectClient 配置。

以下 server.xml 文件示例显示了两个 Open ID Connect 客户机配置，其中配置了 authFilter 以将请求路由到 RP2 配置。 如果显示了签发者声明值为 https://hostname/op1 的请求，那么将选择 RP1 配置，除非 rp2filter 适用于该请求。

<openidConnectClient id="RP1" issuer="https://hostname/op1"  ... />
<openidConnectClient id="RP2" issuer="https://hostname/op2" authFilterRef="rp2filter" .../>

<authFilter id="rp2filter">
  ...
</authFilter>

Open Liberty 根据以下算法选择要用于 JWT 请求的 openidConnectClient 配置。

  1. 如果 openidConnectClient 配置的认证过滤器与请求匹配，请选择该配置。

  2. 如果来自 JWT 的签发者声明仅与一个 openidConnectClient 配置元素中的 issuer 属性匹配，请选择该配置。

  3. 如果来自 JWT 的签发者声明与多个 openidConnectClient 配置元素中的 issuer 属性匹配，请选择第一个匹配的配置。

  4. 如果来自 JWT 的签发者声明与任何 openidConnectClient 配置元素中的 issuer 属性不匹配，请以非确定性方式在所有配置元素之间进行选择。

    声明到主题映射

    如果未定义索赔到标的映射，则适用以下默认规则。

    • 主体集 (sub) 声明用作主体名称和用户的唯一安全名称。

    • 签发人 (iss)声明是默认域，用作主题域。 如果 JWT 中包含 realmName 权利要求，则 realmName 权利要求将被用作主题域，而不是 iss 权利要求。

      您可以通过指定以下 openidConnectClient 元素属性来定制如何将 JWT 声明映射到 Open Liberty 创建的主题。

    • userIdentifier

    • userUniqueIdentifier

    • groupIdentifier

    • realmName

    • realmIdentifier

      如果同时配置了 realmName 和 realmIdentifier 属性，那么 realmName 值优先， realmIdentifier 值将被忽略。

      您可以指定备用声明作为其中任何属性的值。 例如，要将 name 声明用于主体名称而不是 sub 声明，可以指定 userIdentifier="name"。 有关这些属性的更多信息，请参阅 openidConnectClient 元素。

      您还可以通过实施 com.ibm.wsspi.security.oauth.UserCredentialResolver 服务编程接口 (SPI)，以编程方式将 JWT 映射到主题。 有关信息，请参阅 ${wlp.install.dir}/dev/spi/ibm/ 目录中随 Open Liberty 安装一起提供的 Java 文档。

    JWT 受众

    要定义受信任受众的列表，请针对 openidConnectClient 元素配置 audiences 属性。

    有效的 JWT 必须满足以下条件之一。

    • 如果配置了 audiences 属性，则 JWT 中的 aud 索赔值必须是已配置的受众之一。 要忽略受众检查，请将 audiences 属性设置为 ALL_AUDIENCES。

    • 如果未配置 audiences 属性，但 JWT 包含作为有效 URL 的 aud 声明，那么要访问的资源的 URL 必须包含完整的 aud 值作为前缀。

      例如，以下受众是有效的，因为资源 URL 以 JWT 中完整的 aud 索赔值开头。

      • 观众要求："aud":"https://<server>:<port>/something"

      • 资源服务 URL：https://<server>:<port>/something/specific

    资源请求

    如果 Web 客户机将 JWT 发送到请求 Authorization 头中的 Open Liberty 服务器或作为表单编码主体参数，那么不需要额外的服务器配置。 如果在定制请求头字段中发送 JWT ，那么必须将 openidConnectClient 元素的 headerName 属性配置为等于该字段的名称，如以下示例中所示。

    <openidConnectClient id="customHeader"
    ...
    headerName="jwt"
    ...
</openidConnectClient>

    通过此配置， Web 客户机可以在 jwt 头字段上发送令牌，如以下资源请求示例中所示。

    GET /resource HTTP/1.1
Host: server.example.com
jwt: mF_9.B5f-4.1JqM

接受 JWE 格式的令牌

在 V 22.0.0.1 和更高版本中，可以配置 Open Liberty OIDC 客户机以处理 JWE 格式的访问令牌和标识令牌。 相应的 OP 必须支持创建 JWE 访问令牌或标识令牌。 根据 OIDC 提供者文档，使用用于加密内容加密密钥的 OIDC 客户机公用密钥设置 OP。

以下示例显示了用于接受 JWE 格式的 JWT 的样本配置。

<openidConnectClient id="JWE" inboundPropagation="required"
  jwkEndpointUrl="https://example.com/jwtserver/jwk" signatureAlgorithm="RS256"
  issuerIdentifier="https://idp.example.com:8020/jwt"
  keyManagementKeyAlias="privateKeyAlias"
  accessTokenCacheEnabled="true"
  accessTokenCacheTimeout="10s"
  tokenReuse="true">
</openidConnectClient>

  • 必须将 keyManagementKeyAlias 属性设置为用于解密 JWE 格式令牌的内容加密密钥的密钥管理密钥的专用密钥别名。 此密钥必须存在于为 OIDC 配置的密钥库中。

  • 此示例指定可选属性以配置 RP 来高速缓存入站传播的访问令牌验证结果。

    • accessTokenCacheEnabled 属性允许对验证结果进行高速缓存。

    • accessTokenCacheTimeout 属性指定对已认证的主体集进行高速缓存的持续时间。

    • tokenReuse 属性指定 OpenID Connect 客户机必须对包含 jti 声明的 JWT 的结果进行高速缓存。

如果使用 Social Login 功能部件管理 OIDC 的单点登录 (SSO) ，那么可以启用该功能部件以处理 JWE 格式的令牌。 在 oidcLogin 元素上指定 keyManagementKeyAlias 属性。 将值设置为用于解密 JWE 格式令牌的内容加密密钥的密钥管理密钥的专用密钥别名。 有关更多信息，请参阅 OIDC Social Login。

配置第三方 OpenID Connect 提供者

要将 Liberty OIDC 客户端配置为使用第三方 OIDC 提供程序，如 Microsoft Azure 或 Google ，必须配置以下属性。 通过调用 OP 发现端点来获取这些属性值，该端点在通过将 /.well-known/openid-configuration 字符串添加到签发者 URI 而形成的路径上提供 JSON 文档。 例如，对于 Google，此文档可从 https://accounts.google.com/.well-known/openid-configuration URL 获取。

  • jwkEndpointUrl

    将此属性设置为由发现文件中的 jwks_uri 值定义的 OP JSON Web 密钥集文档的 URL。

  • issuerIdentifier

    将此属性设置为发现文件中定义的签发人。 未包含此值作为 iss 声明的标识令牌会被拒绝。

  • signatureAlgorithm="RS256"

    Liberty OIDC 客户机缺省签名算法为 HS256。

  • userIdentityToCreateSubject

    将此属性设置为供应商 ID 令牌使用的权利要求名称，该名称代表用户的唯一标识符。

  • groupIdentifier

    将属性设置为代表用户组员资格或角色的声明名称。

以下示例显示了用于作为 OIDC 提供者连接到 Google 的样本配置。

<openidConnectClient id="sample client"
    clientId="ebb996943plm4450bdb674c211a9fab0"
    clientSecret="g12ciZL1zsVYk6yXTMa2iopa8TBwC178CGq42mdJCALYCdQT1kE1ecB010GE"
    discoveryEndpointUrl="https://accounts.google.com/.well-known/openid-configuration"
    jwkEndpointUrl="https://www.googleapis.com/oauth2/v3/certs"
    issuerIdentifier="https://accounts.google.com"
    signatureAlgorithm="RS256"
    userIdentityToCreateSubject="email"
    groupIdentifier="aud">
</openidConnectClient>

支持多个 OpenID 连接提供程序

通过创建多个 openidConnectClient 元素并设置认证过滤器以将入局认证请求路由到每个配置，可以将 Open Liberty 作为 OIDC 客户机配置到多个 OPs。 每个 openidConnectClient 元素定义一个与一个 OIDC 提供者的 SSO 关系。 使用 authFilterRef 属性来引用认证过滤器，以指定为每个 openidConnectClient 元素配置的认证过滤器。

在以下示例中，包含 /mywebapp/members URL 模式的认证请求将过滤为 RP1 配置所引用的 OP。 包含 /mywebapp/guests URL 模式的认证请求将过滤为 RP2 配置所引用的 OP。

<openidConnectClient id="RP1"
    authFilterRef="authFilter1"
    clientId="RP1"
    clientSecret="{xor}LDo8LTor"
    authorizationEndpointUrl="https://server.example.com:443/oidc/endpoint/OidcConfigSample/authorize"
    tokenEndpointUrl="https://server.example.com:443/oidc/endpoint/OidcConfigSample/token">
</openidConnectClient>

<openidConnectClient id="RP2"
    authFilterRef="authFilter2"
    clientId="RP2"
    clientSecret="{xor}DLo8LTor"
    authorizationEndpointUrl="https://server.example2.com:443/oidc/endpoint/OidcConfigSample/authorize"
    tokenEndpointUrl="https://server.example2.com:443/oidc/endpoint/OidcConfigSample/token">
</openidConnectClient>

<authFilter id="authFilter1">
    <requestUrl
        id="myUrlFilter"
        urlPattern="/mywebapp/members"
        matchType="contains" />
</authFilter>

<authFilter id="authFilter2">
    <requestUrl
        id="myUrlFilter2"
        urlPattern="/mywebapp/guests"
        matchType="contains" />
</authFilter>

有关更多信息，请参阅 身份验证筛选器。

使用专用密钥 JWT 进行客户机认证

OpenID Open Liberty 中的 Connect 客户机支持使用 OpenID Connect 令牌端点的 private_key_jwt 客户机认证方法。 OpenID Connect 客户机需要向 OpenID Connect 提供程序提供认证数据以访问提供程序的令牌端点。 客户机可以使用多种不同的方法进行认证，但其中大多数方法都需要客户机密钥。 private_key_jwt 认证方法允许客户机使用非对称密钥来创建已签名的 JWT 以进行认证，而不是客户机密钥。 通过使用此认证方法， Open Liberty 中的 OpenID Connect 客户机不再需要具有客户机密钥。

服务器管理员可以通过对 tokenEndpointAuthMethod 属性以及 openidConnectClient 元素中的新 tokenEndpointAuthSigningAlgorithm 和 keyAliasName 属性使用 private_key_jwt 选项来启用此功能。

<openidConnectClient id="myOidcClientUsingPrivateKeyJwt"
    tokenEndpointAuthMethod="private_key_jwt"
    tokenEndpointAuthSigningAlgorithm="E512"
    keyAliasName="privatekeyaliasES512"
    ...
/>

tokenEndpointAuthSigningAlgorithm 属性指定用于对用于客户机认证的 JWT 进行签名的签名算法。 keyAliasName 属性指向用于对 JWT 进行签名的专用密钥。 专用密钥必须存在于 OpenID Connect 客户机配置中的 sslRef 属性所指定的密钥库中。 OpenID 支持专用密钥 JWT 客户机认证的 Connect 提供者通常为管理员提供用于上载其相应公用密钥的接口。

有关 private_key_jwt 客户机认证的更多信息，请参阅 OpenID Connect 核心规范 和 RFC 7523。

禁用 LTPA cookie

Open Liberty OIDC 客户机在处理标识令牌后自动创建 SSO 令牌。 通过为 openidConnectClient 元素指定 disableLtpaCookie 属性，可以将服务器配置为不创建 SSO 令牌。

<openidConnectClient id="nocookie"
    ...
    disableLtpaCookie="true"
    ...
</openidConnectClient>

将此属性设置为 true时， Open Liberty OIDC 客户机仅接受先前已向配置的 OP 认证的认证请求。 认证会话生存期限制为来自该 OP 的标识令牌的生存期。

接受 OAuth 2.0 不记名访问令牌，而不重定向到 OpenID Connect 提供者

您可以将 OIDC 客户端配置为可选择接受有效的 OAuth 2.0 承载访问令牌作为身份验证令牌，而无需将请求重定向到 OIDC 提供程序。 通过此配置，如果请求包含有效的 OAuth 2.0 不记名访问令牌，那么 Open Liberty OIDC 客户机会自动对其进行验证，并根据令牌验证结果创建已认证的主体集。 如果请求不包含访问令牌或访问令牌无效，则 Open Liberty OIDC 客户端会继续将用户重定向到 OIDC 提供商。 该函数可使 Open Liberty 服务器像 RESTful 客户端一样为浏览器客户端和非浏览器客户端提供服务。

<openidConnectClient id="nocookie"
    ...
    inboundPropagation="supported"
    ...
</openidConnectClient>

修改上下文根目录

如果您的托管环境不允许访问 /oidcclient 上下文根目录，您可以通过配置 oidcClientWebapp 元素来修改上下文根目录。 缺省情况下， Open Liberty OIDC 客户机重定向 servlet 侦听 /oidcclient 上下文根，并且重定向 URL 格式为 https://host_name: ssl_port/oidcclient/redirect/configuration_ID。 如果无法使用此上下文根，那么可以设置其他上下文根。

例如，如果您的主机环境要求您使用 /example/openid 上下文根，请在 server.xml 文件中添加以下元素。

<oidcClientWebapp contextPath="/example/openid" />

生成的重定向 URL 格式为 https://host_name: ssl_port/example/openid/redirect/configuration_ID。

将请求参数传递到 OpenID Connect

要向 OP 提供额外的授权参数，请配置 authzParameter 元素。 例如，以下配置属性将 api_key 和 account 参数传递至授权端点。

<authzParameter name="api_key" value="567890" />
<authzParameter name="account" value="123456" />

要为令牌端点配置参数，请使用 tokenParameter 元素。

检查访问令牌以获取用户和组信息

如果在标识令牌中找不到 userIdentifier 或 groupIdentifier 值，那么 Open Liberty 还可以检入访问令牌和 UserInfo 端点。

将 tokensOrderToFetchCallerClaims 属性设置为 AccessToken IDToken Userinfo，如以下示例中所示。

<openidConnectClient tokensOrderToFetchCallerClaims="AccessToken IDToken Userinfo"  userIdentifier="unique_name" groupIdentifier="aud" ... />

在此示例中， Open Liberty 首先在访问令牌中检查 userIdentifier 和 groupIdentifier 值，然后检查标识令牌，最后调用 UserInfo 端点。 如果在这些位置中的任何位置都找不到这些值，那么将返回未经授权的响应代码。

如果未指定 tokensOrderToFetchCallerClaims 属性，那么缺省值为 IDToken ，并且 Open Liberty 仅检查标识令牌以获取用户和组信息。
