除了常规 InstantOn 先决条件外， Open Liberty InstantOn 还受某些限制。 例如，必须运行早期启动代码或依赖于某些 Liberty 功能部件的应用程序可能需要修改才能使用 InstantOn。

有关 InstantOn 先决条件的更多信息，请参阅 运行时和主机构建系统先决条件。

Jakarta Transactions 配置限制

Open Liberty Transaction Manager 对 InstantOn 的支持在复原应用程序进程时具有有关配置更新的限制。 事务管理器 的配置属性必须在 InstantOn 检查点和复原之间保持常量。 此限制仅适用于直接通过 transaction 服务器配置元素 (例如， recoveryGroup 和 recoverIdentity) 指定的配置属性。 这些配置属性的值不得在检查点和复原之间更改。

此限制意味着 云环境中的事务恢复 无法按设计工作，因为 recoverIdentity 无法通过类似于以下 transaction 配置示例的内容进行参数化。 此示例为应用程序的每个实例提供了唯一的 recoverIdentity :

<transaction
  ...
  recoveryGroup="peer-group-name"
  recoveryIdentity="${HOSTNAME}${wlp.server.name}"
  ...
/>

Jakarta Transaction before Checkpoint

Open Liberty InstantOn 不允许在对应用程序进程执行检查点之前开始事务。 如果应用程序具有尝试启动事务的 早期启动代码 ，那么此方案是可能的。 请考虑以下 Servlet:

@WebServlet(urlPatterns = { "/StartupServlet" }, loadOnStartup = 1)
public class StartupServlet extends HttpServlet {
    @Override
    public void init() {
        UserTransaction ut = UserTransactionFactory.getUserTransaction();
        try {
            ut.begin();
            ...
            ut.commit();
        } catch (Exception e) {
            // something went wrong
        }
    }

}

此 Servlet 示例使用 loadOnStartup = 1 属性。 将此属性与 afterAppStart 选项配合使用时， servlet 会在检查点之前初始化。 运行时检测到此冲突并记录以下消息:

[WARNING ] WTRN0155W: An application began or required a transaction during the server checkpoint request. The following stack trace for this thread was captured when the transaction was created:

此警告后跟一个堆栈跟踪，该堆栈跟踪可帮助标识尝试启动事务的应用程序代码。 然后，服务器将无法执行检查点操作，并且将记录以下错误:

WTRN0154E: The server checkpoint request failed because the transaction service is unable to begin a transaction.

您可以通过使用 beforeAppStart 选项或修改组件以不使用早期启动代码来避免此故障。 在此示例中，该修改将除去 loadOnStartup = 1 属性。

访问 MicroProfile 配置过早

如果应用程序具有早期启动代码，并且您正在使用 afterAppStart 选项，那么在对应用程序进程执行检查点之前，可能会使用 MicroProfile Config 中的配置值对其进行注入。 如果在应用程序映像容器运行时此类配置值发生更改，那么应用程序可能会使用执行应用程序进程检查点时设置的旧值。

Open Liberty 运行时检测到此情境，并在运行应用程序容器映像时记录一条警告消息，指示配置值已更改。 以下示例使用缺省值设置为 theDefault的 example_config 配置键。 发生检查点时，环境配置源不可用于填充 MicroProfile 配置值。 如果配置的此 @Inject 注释包含在执行检查点之前创建并使用的 CDI Bean 中，那么将注入 theDefault 的值。

    @Inject
    @ConfigProperty(name = "example_config", defaultValue = "theDefault")
    String exampleConfig;

运行 InstantOn 应用程序容器映像时，可以使用环境变量 EXAMPLE_CONFIG 来提供更新的值。 运行时检测到此值并记录以下消息:

[WARNING ] CWWKC0651W: The MicroProfile configuration value for the key example_config has changed since the checkpoint action completed on the server. If the value of the key changes after the checkpoint action, the application might not use the updated value.

在此情况下，请使用 beforeAppStart 检查点选项。 另一个选项是使用 动态 ConfigSource。 可以通过将 Provider<String> 类型用于 exampleConfig 变量来修改先前的示例，以使用动态 ConfigSource :

    @Inject
    @ConfigProperty(name = "example_config", defaultValue = "theDefault")
    Provider<String> exampleConfig;

对 Provider<String> 的 get() 方法的每个调用都会返回 ConfigProperty 注释的当前值。 此行为允许应用程序在 InstantOn 应用程序容器运行期间复原应用程序进程时访问更新后的配置值。

过早注入 DataSource

如果应用程序具有早期启动代码，并且您正在使用 afterAppStart 选项，那么在对应用程序进程执行检查点之前，可能会将其注入 DataSource 。 在云环境中，在运行应用程序映像容器时，可能需要更改 DataSource 的配置。 请考虑以下 Servlet 示例:

@WebServlet(urlPatterns = "/ExampleServlet", loadOnStartup = 1)
public class ExampleServlet extends HttpServlet {
    @Resource(shareable = false)
    private DataSource exampleDataSource;
    ...
}

此 Servlet 示例使用 loadOnStartup = 1 属性。 使用 afterAppStart 选项时，此属性会在检查点之前初始化 servlet。 将应用程序部署到云时，可能需要配置与 DataSource 相关的部署信息。 请考虑以下 Open Liberty server.xml 配置。

  <!-- these values are place holders so we don't have to have the env set before checkpoint -->
  <variable name="DB2_DBNAME" defaultValue="placeholder" />
  <variable name="DB2_HOSTNAME" defaultValue="placeholder" />
  <variable name="DB2_PASS" defaultValue="placeholder" />
  <variable name="DB2_PORT" defaultValue="45000" />
  <variable name="DB2_PORT_SECURE" defaultValue="45001" />
  <variable name="DB2_USER" defaultValue="placeholder" />


  <dataSource id="DefaultDataSource">
    <jdbcDriver libraryRef="DB2Lib"/>
    <properties.db2.jcc
      databaseName="${DB2_DBNAME}" serverName="${DB2_HOSTNAME}" portNumber="${DB2_PORT}"
      downgradeHoldCursorsUnderXa="true"/>
    <containerAuthData user="${DB2_USER}" password="${DB2_PASS}"/>
    <recoveryAuthData user="${DB2_USER}" password="${DB2_PASS}"/>
  </dataSource>

此配置将占位符值用于诸如数据库名称，主机名，端口，用户和密码之类的内容。 此配置允许使用环境变量值或其他配置机制更新值，如 配置在 Kubernetes中运行的微服务中所述。 这些配置不得硬编码到应用程序映像中，并且必须能够在将应用程序部署到云时进行更新。

如果在检查点之前使用 DataSource 注入应用程序，并且 DataSource 的配置发生更改，那么当使用更新后的配置运行 InstantOn 应用程序容器映像时，将重新启动应用程序。 您可以通过使用 beforeAppStart 选项或通过将组件修改为不作为早期启动代码来避免此场景。 在此示例中，该修改将除去 loadOnStartup = 1 属性。

访问 MicroProfile 配置属性，检查点没有缺省值

使用未在任何配置源中设置缺省值的配置属性注入的应用程序可能会导致检查点期间发生错误。 本部分提供了迂到的常见错误的解决方案。

可以静态或动态地将配置属性引入到应用程序中，在任何一种情况下，都可以将该属性声明为可选属性。 以下示例显示了注入静态，静态-可选，动态和动态-可选配置属性的方法。

  @Inject
  @ConfigProperty(name = "static_config")
  String staticConfig;

  @Inject
  @ConfigProperty(name = "static_optional_config")
  Optional<String> staticOptionalConfig;

  @Inject
  @ConfigProperty(name = "dynamic_config")
  Provider<String> dynamicConfig;

  @Inject
  @ConfigProperty(name = "dynamic_optional_config")
  Provider<Optional<String>> dynamicOptionalConfig;

如果在检查点期间在现有配置源中找不到任何值，那么注入的 static_config 属性会导致类似以下示例的错误:

SRCFG02000: Failed to Inject @ConfigProperty for key static_config into io.example.Example.staticConfig since the config property could not be found in any config source.

可以通过下列其中一种方法为配置键提供缺省值来避免此错误:

在 @ConfigProperty 注释上指定缺省值
  @Inject
  @ConfigProperty(name = "static_config", defaultValue = "defaultValue")
  String staticConfig;
在应用程序 META-INF/microprofile-config.properties 资源中指定缺省值
  static_config=defaultValue
在 server.xml` 文件的 variable 元素中指定缺省值
  <variable name="static_config" defaultValue="defaultValue" />

如果未设置缺省值，那么仍可以通过使用 static_optional_config， dynamic_config或 dynamic_optional_config 属性注入配置来避免先前的错误。 但是，如果将检查点选项与 @ApplicationScoped 的 CDI Bean 配合使用，并且在应用程序启动期间过早访问 dynamic_config ，那么可能会发生以下错误:

java.util.NoSuchElementException: SRCFG00014: The config property dynamic_config is required but it could not be found in any config source.

同样，过早访问 static_optional_config 和 dynamic_optional_config 可能会导致以下错误:

java.util.NoSuchElementException: No value present

因此，为了避免这些错误，最好为注入的配置属性设置缺省值，因为在应用程序启动期间，可以过早访问可选的动态配置。 此外，如果 @ConfigProperty 注入站点未使用动态配置，那么在复原时不会更新在检查点之前注入到应用程序范围的 Bean 中的任何缺省值。 有关更多信息，请参阅 过早访问 MicroProfile 配置

使用产品扩展，用户功能部件或 InstantOn 不支持的功能部件

InstantOn 仅支持 Open Liberty 功能部件的子集，如 Open Liberty InstantOn 支持的功能部件中所述。 在 InstantOn 的受支持功能集之外启用的任何公共功能都会导致检查点失败，并显示类似以下示例的错误消息:

CWWKC0456E: A checkpoint cannot be taken because the following features configured in the server.xml file are not supported for checkpoint: [usr:exampleFeature-1.0]

对于 InstantOn不支持的任何已配置功能，都会发生此错误。 此限制包括 Liberty 产品扩展和 Liberty 用户功能部件。

使用 bootstrap.properties 文件更新配置

运行 InstantOn 应用程序容器映像时，不会读取 bootstrap.properties 文件。 在运行 InstantOn 应用程序容器映像时必须能够配置的值必须来自备用源。 例如，您可以使用环境变量或其他配置机制，如 配置在 Kubernetes中运行的微服务所述。

不支持 Java SecurityManager

如果 Open Liberty 配置为与 SecurityManager一起运行，那么 InstantOn 会在检查点期间检测到此配置，并失败并显示以下消息:

CWWKE0958E: The server checkpoint request failed because the websphere.java.security property was set in the bootstrap.properties file. This property enables the Java Security Manager and is not valid when a server checkpoint occurs.

更新 JVM 选项

InstantOn 不支持在复原 InstantOn 应用程序进程时更改 jvm.options 。 必须在 InstantOn 容器映像构建期间定义需要为 JVM 进程设置的任何 JVM 选项。

IBM Semeru JVM 对于使用 OPENJ9_RESTORE_JAVA_OPTIONS 环境变量在复原时设置 JVM 选项的支持有限。 有关更多信息，请参阅 Java Checkpoint/Restore In Userspace (CRIU) 支持 文档。

SELinux 限制

如果 SELinux 方式设置为 enforcing，那么当您在映像模板 Dockerfile 或 Containerfile中使用 checkpoint.sh 脚本 时， SELinux 可能会阻止 InstantOn 执行应用程序进程的检查点。 如果禁用了 virt_sandbox_use_netlink SELinux 设置，那么将阻止必需的 netlink Linux 系统调用。 此块阻止 InstantOn 在容器映像构建期间执行应用程序进程的检查点。 Open Liberty InstantOn 会检测到此限制，并记录以下消息:

CWWKE0962E: The server checkpoint request failed. The following output is from the CRIU /logs/checkpoint/checkpoint.log file that contains details on why the checkpoint failed.
Warn  (criu/kerndat.c:1103): $XDG_RUNTIME_DIR not set. Cannot find location for kerndat file
Error (criu/libnetlink.c:84): Can't send request message: Permission denied
..
Error (criu/cr-dump.c:2099): Dumping FAILED.
CWWKE0963E: The server checkpoint request failed because netlink system calls were unsuccessful. If SELinux is enabled in enforcing mode, netlink system calls might be blocked by the SELinux "virt_sandbox_use_netlink" policy setting. Either disable SELinux or enable the netlink system calls with the "setsebool virt_sandbox_use_netlink 1" command.

要解决此限制，可以使用 setsebool virt_sandbox_use_netlink 1 命令启用 virt_sandbox_use_netlink SELinux 设置或禁用 SELinux enforcing 方式。 解决此问题的另一个选项是使用 三步过程 来构建 InstantOn 映像。 三步过程需要使用 --privileged 容器，该容器将 netlink 系统调用的访问权授予执行应用程序进程检查点的正在运行的容器。

Yama Linux 安全模块限制

如果使用下列其中一种方式配置了 Yama ，那么 InstantOn 无法在运行的容器中对应用程序进程进行检查点或复原:

  • 2 -仅管理连接

  • 3 -无连接

存在此配置时， /logs/checkpoint/restore.log 包含以下错误:

Error (criu/arch/x86/kerndat.c:178): 32: ptrace(PTRACE_TRACEME) failed: Operation not permitted

对于 InstantOn 检查点和复原工作，必须使用下列其中一种方式配置 Yama:

  • 0 -经典 ptrace 许可权

  • 1 -受限 ptrace

以下受支持的公共云 Kubernetes 服务将 Yama 的缺省值设置为 1 方式，这允许 InstantOn 在缺省情况下进行检查点和复原:

  • Amazon Elastic Kubernetes Service (EKS)

  • Azure Kubernetes 服务 (AKS)

访问 Linux 系统调用

如 必需的 Linux 系统调用中所述， CRIU 需要多个 Linux 系统调用来复原应用程序进程。 当您使用 InstantOn时，此需求可能需要额外配置以将必需的系统调用授予正在运行的容器。

以下示例是阻止访问特定系统调用时记录到 /logs/checkpoint/restore.log 文件的错误。

已阻止 clone3 系统调用
Error (criu/kerndat.c:1377): Unexpected error from clone3: Operation not permitted
阻塞到 ptrace 系统调用
Error (criu/arch/x86/kerndat.c:178): 28: ptrace(PTRACE_TRACEME) failed: Operation not permitted
阻塞到 vmsplice 系统调用
Error (criu/pipes.c:184): 0x4c11a: Error splicing data: Operation not permitted

缺省情况下，受支持的公共云 Kubernetes Service 环境当前允许 CRIU 使用必需的系统调用。 使用以下云提供者时，不需要其他配置:

  • Amazon Elastic Kubernetes Service (EKS)

  • Azure Kubernetes 服务 (AKS)

在没有必需 Linux 功能的情况下运行

如果未授予必需的 Linux 功能 ，那么在检查点和复原期间将发生错误。 如果未授予检查点所需的功能，那么在 InstantOn 容器映像构建期间将发生以下错误:

Can't exec criu swrk: Operation not permitted
Can't read request: Connection reset by peer
Can't receive response: Invalid argument
[ERROR   ] CWWKC0453E: The server checkpoint request failed with the following message: Could not dump the JVM processes, err=-70

Operation not permitted 消息指示未授予必需的 Linux 功能。 如果您正在使用 checkpoint.sh 脚本，那么 RUN checkpoint.sh 指令期间将发生以下错误:

Error: building at STEP "RUN checkpoint.sh afterAppStart": while running runtime: exit status 74

要避免此错误，请授予容器映像构建 CHECKPOINT_RESTORE， SYS_PTRACE和 SETPCAP Linux capabilities。 如果使用 三步过程 来构建容器映像，请确保正在运行检查点步骤的容器是 --privileged 容器。

如果未授予复原所需的功能，那么当您尝试运行 InstantOn 应用程序容器映像时，会发生以下错误:

/opt/ol/wlp/bin/server: line 1430: /opt/criu/criu: Operation not permitted
CWWKE0961I: Restoring the checkpoint server process failed. Check the /logs/checkpoint/restore.log log to determine why the checkpoint process was not restored. Launching the server without using the checkpoint image.

Operation not permitted 消息指示未授予复原所需的 Linux 功能。
