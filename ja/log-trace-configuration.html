ログおよびトレースの構成

Open Liberty には、アプリケーションとランタイムによって書き込まれるメッセージを処理する統合ロギング コンポーネントがあり、FFDC (First Failure Data Capture) 機能を提供します。 System.out、 System.err、または java.util.logging.Logger ストリームを使用してアプリケーションによって書き込まれるロギング・データは、サーバー・ログに結合されます。

サーバーには、以下の 4 つの 1 次ログ・ファイルがあります。

  • console.log -このファイルは、 server start コマンドによって作成されます。 このストリームには、JVMプロセスからリダイレクトされた標準出力と標準エラー・ストリームが含まれる。 このコンソール出力は、人間が読みやすいようにフォーマットされており、自動ログ分析に役立つ情報が不足しています。 console.log ファイルは、サーバーが再始動されるとリセットされます。

  • messages.log - このファイルは、ロギング・コンポーネントによって書き込まれる、または取り込まれるすべてのメッセージを含みます。 このファイルに書き込まれるすべてのメッセージには、メッセージのタイムスタンプやメッセージを書き込んだスレッドのIDなどの追加情報が含まれている。 このファイルは、自動ログ分析に適しています。 このファイルには、JVM プロセスに直接書き込まれるメッセージは含まれません。

  • trace.log - このファイルはトレースを有効にした場合のみ作成されます。 これには、 messages.log ファイルのすべての内容に加えて、有効なトレースが含まれます。 このファイルには、JVM プロセスに直接書き込まれるメッセージは含まれません。

  • verbosegc.XXX.log -これらのファイルには JVM からの詳細なガーベッジ・コレクション出力が含まれており、Java 実装が IBM Java または IBM Semeru Runtimes のいずれかである場合にデフォルトで作成されます。 最大 10 個のローリング・ログ・ファイルがログ・ディレクトリーに作成され、1 ファイル当たりの GC サイクル数は 1024 になります。ここで、 XXX はログ・ファイルのシーケンス番号を表します。 カスタム構成は、デフォルトの動作よりも優先されます。 詳細ガーベッジ・コレクション・ログを無効にするには、 server.env ファイルに VERBOSEGC=false を追加します。 詳しくは、 冗長ガーベッジ・コレクション・ログを参照してください。

user.timezone JVM プロパティーを使用して、アプリケーション環境の時間帯を設定できます。 jvm.propertiesファイルにuser.timezoneプロパティを設定する。 更新されたタイム・ゾーン情報が messages.log ファイルおよび trace.log ファイルに表示されます。 次の例は、タイムゾーンの設定方法を示している。

-Duser.timezone=time_zone_code

ロギング構成

ロギング・コンポーネントは、サーバー構成を通じて制御できます。 ロギング・コンポーネントは、 logging エレメントを使用して server.xml ファイル内で完全に構成できます。 ただし、ロギングは server.xml ファイルが処理される前に初期化されるため、 server.xml ファイルを使用してロギングを構成すると、後のログ構成とは異なるログ構成を使用する早期ログ項目が作成される可能性があります。 この問題を回避するために、 boostrap.properties ファイルおよび場合によっては環境変数を使用して、ロギング構成の多くを提供することができます。 bootstrap.propertiesとserver.xmlファイルの両方で、同じ構成プロパティに異なる値を指定することは避けてください。 ロギング構成設定について詳しくは、「 ソース別の構成設定 」の表を参照してください。

ログ・ファイル・ストレージ管理

console.log ファイルは、プロセス stdout および stderr ストリームをファイルにリダイレクトすることによって作成されます。 その結果、 Open Liberty は、 messages.log ファイルに対して提供するのと同じレベルの管理 (ログ・ロールオーバーなど) を提供できません。 console.log ファイルのサイズが心配な場合は、コンソール・ログを OFF に設定し、代わりに messages.log ファイルを使用することができます。 ロギング・コンポーネントによって書き込まれるか取り込まれるすべてのメッセージは、 messages.log ファイルに書き込まれます。 メッセージ・ログでファイル・ロールオーバーを構成できます。

ただし、コンソール・ログ・レベルを OFFに設定した場合でも、基礎となる JVM からのメッセージは console.log ファイルに書き込まれます。 同じ原則が、ネイティブ標準出力および標準エラー・ストリームに書き込まれるすべてのメッセージに適用されます。 Open Liberty ロギング・コンポーネントは、JVM プロセスによって直接書き込まれるメッセージ ( -verbose:gc 出力など) の宛先をキャプチャーまたは制御しません。 これらのメッセージは、アクティブな JVM 設定に従ってファイル・システムに経路指定され、 messages.log ファイルには表示されません。

ロギング・コンポーネントがコンソール・ログに書き込まれないようにし、100 Mb で 3 回ロールオーバーするように messages.log ファイルを構成するには、以下の構成を使用します。

com.ibm.ws.logging.max.file.size=100
com.ibm.ws.logging.max.files=3
com.ibm.ws.logging.console.log.level=OFF
com.ibm.ws.logging.copy.system.streams=false

ログ・フォーマット

3 つの Open Liberty 1 次ログ・ファイルのそれぞれに、デフォルトのログ・フォーマットと、ロギング構成で指定できるいくつかの代替フォーマットがあります。 ログごとに選択する形式は、アプリケーションとインフラストラクチャーのニーズによって異なります。 例えば、メッセージ・ログがログ分析プラットフォームによってコンシュームされる場合、メッセージ・ログを JSON 形式で出力することを選択できます。 あるいは、すべてのログ・ファイルにわたって一貫性のあるログ・フォーマットが必要な場合は、3 つすべてのログに対して TBASIC フォーマットを選択することもできます。

  • console.log ファイルのデフォルト・フォーマットは DEVです。

  • messages.log ファイルのデフォルト・フォーマットは SIMPLEです。

  • trace.log ファイルのデフォルト・フォーマットは ENHANCEDです。

以下の例は、使用可能な各ログ・フォーマットの出力例を示し、各フォーマットが使用可能なログ ( DEV、 SIMPLE、 JSON、 TBASIC、 ENHANCED、および ADVANCED) をリストしています。

DEV (コンソール・ログ)

DEV 形式のメッセージは、以下のパターンを使用します。

[log level] <message_ID>: <message>

次の例はDEV形式のメッセージです：

[AUDIT] CWWKF0011I: The defaultServer server is ready to run a smarter planet. The defaultServer server started in 7.967 seconds.
SIMPLE (コンソールおよびメッセージ・ログ)

SIMPLE 形式のメッセージは、以下のパターンを使用します。

[time stamp] <thread ID> <fully qualified className> <log level attribute> <message ID>: <message>

次の例はSIMPLE形式のメッセージです：

[18/10/21 14:49:26:246 EDT] 0000003e com.ibm.ws.kernel.feature.internal.FeatureManager            A CWWKF0011I: The defaultServer server is ready to run a smarter planet. The defaultServer server started in 7.844 seconds

この例では、 CWWKF0011I メッセージ ID コードの前にある A 属性は、メッセージ・ログ・レベルが AUDITであることを示しています。 詳しくは、 メッセージ・ログ・レベルの属性を参照してください。

JSON (コンソール・ログおよびメッセージ・ログ)

JSON 形式のメッセージには、メッセージのさまざまなコンポーネントを識別するフィールド名が含まれています。 次の例はJSON形式のメッセージです：

{"type":"liberty_message","host":"HOST","ibm_userDir":"/wlp\/usr\/","ibm_serverName":"defaultServer","message":"CWWKF0011I: The defaultServer server is ready to run a smarter planet. The defaultServer server started in 7.967 seconds.","ibm_threadId":"0000003e","ibm_datetime":"2021-10-18T14:50:58.159-0400","ibm_messageId":"CWWKF0011I","module":"com.ibm.ws.kernel.feature.internal.FeatureManager","loglevel":"AUDIT","ibm_sequence":"1634583058159_0000000000009"}

詳細については、JSONロギングを参照してください。

TBASIC (コンソール、メッセージ、およびトレース・ログ)

TBASIC 形式のメッセージは、以下のパターンを使用します。

[time stamp] <thread ID> <fully qualified className> <log level attribute> <message ID>: <message>

次の例はTBASIC形式のメッセージです：

[18/10/21 14:47:19:718 EDT] 0000003e FeatureManage A   CWWKF0011I: The defaultServer server is ready to run a smarter planet. The defaultServer server started in 7.718 seconds.

この例では、 CWWKF0011I メッセージ ID コードの前にある A 属性は、メッセージ・ログ・レベルが AUDITであることを示しています。 ログ・レベル属性について詳しくは、 メッセージ・ログ・レベル属性を参照してください。

TBASICについて詳しくは、 TBASIC ログ・フォーマットを参照してください。

ENHANCED (トレース・ログ)

ENHANCED 形式のメッセージには、特定のトレース・メッセージのログ・レコードからのコンポーネントであるフィールド名が含まれます。これにより、トレース・メッセージに関する詳細が提供されます。 以下の詳細が含まれています。

  • メッセージ・タイプ

  • 日付と時刻

  • messageID

  • 完全修飾クラス名

  • 重大度レベル

  • ログ・レベル

  • メソッド名

  • クラス名

  • レベル値

  • スレッド名

  • correlationID (または product、component、message、および objectID)

SIMPLE または TBASIC 形式のメッセージには、メッセージ ID コードの直前に、以下のいずれかのメッセージ・ログ・レベル属性が含まれます。

TBASIC ログ・フォーマット

TBASIC ロギング・フォーマットは、 Open Liberty コンソール、メッセージ、およびトレース・ログ・ファイル全体で一貫性のあるログ・フォーマットを提供します。 トレース・ログには、 BASIC ロギング・フォーマットが存在します。 ただし、トレース・ログとコンソール・ログおよびメッセージ・ログの出力は異なります。 TBASIC フォーマットは、トレース・ログの BASIC オプションと一致する、トレース・ログ、コンソール・ログ、およびメッセージ・ログの一貫性のあるフォーマット・オプションを提供します。 TBASIC フォーマットは、 BASIC オプションの別名として機能します。

以下の例に示すように、 bootstrap.properties ファイルに TBASIC 形式を指定できます。

com.ibm.ws.logging.message.format=tbasic
com.ibm.ws.logging.console.format=tbasic
com.ibm.ws.logging.trace.format=tbasic

以下の例に示すように、メッセージおよびコンソール・ログの TBASIC ログ・フォーマットを環境変数として server.env ファイルに指定することもできます。

WLP_LOGGING_MESSAGE_FORMAT=TBASIC
WLP_LOGGING_CONSOLE_FORMAT=TBASIC

JSON ロギング

JSON 形式でログを生成することで、ログの構文解析を簡素化できます。 JSON は、多くのログ分析ツールが、フォーマット固有の構文解析命令を必要とせずに取り込むことができる自己記述型フォーマットです。 bootstrap.properties ファイルを編集するか、環境変数を指定することによって、JSON 形式でログを生成するように Open Liberty ログを構成できます。 以下の 2 つの例は、これらの各オプションの構成を示しています。

  • bootstrap.properties ファイルで JSON ロギングを構成します。

    com.ibm.ws.logging.message.format=json
com.ibm.ws.logging.message.source=message,trace,accessLog,ffdc,audit

  • server.env ファイル内の環境変数を使用して JSON ロギングを構成します。

    WLP_LOGGING_MESSAGE_FORMAT=json
WLP_LOGGING_MESSAGE_SOURCE=message,trace,accessLog,ffdc,audit

構成可能な JSON フィールド名

ログがJSON形式の場合、jsonFieldMappings属性を使ってデフォルトのフィールド名を新しいフィールド名に置き換えることができます。 同じロギング構成内の他のサーバーが、 Open Liberty のデフォルト名とは異なるフィールド名を使用している場合、デフォルトのフィールド名の置換が必要になることがあります。 例えば、 Open Liberty メッセージは message フィールド名によって参照されますが、別のコンテナー内のメッセージは logというフィールドに存在する可能性があります。 この場合、メッセージの 2 つの異なる視覚化がダッシュボード上のログに表示されます。 他のログと一致するように Open Liberty 出力フィールド名を変更すると、それらを同じ視覚化で表示できます。 以下の例は、JSON フィールドを名前変更するためのサンプル構成を示しています。

  • 新規フィールド名を構成するには、 server.env ファイルに以下の環境変数を含めることができます。

    WLP_LOGGING_JSON_FIELD_MAPPINGS=loglevel:level

    この例では、 loglevel フィールド名は level フィールド名に置き換えられます。

  • 特定のソースのフィールド名を構成するには、 server.env ファイルに以下の環境変数を含めることができます。

    WLP_LOGGING_JSON_FIELD_MAPPINGS=message:message:log

    この例では、 message フィールド名は、メッセージ・ログ内の log フィールド名で置き換えられます。

ログからフィールドを省略するには、以下の例に示すように、フィールド名を置換なしで指定する：

WLP_LOGGING_JSON_FIELD_MAPPINGS=defaultFieldName:

複数のフィールドの名前を変更したり、除外したりするには、フィールド名マッピングのコンマ区切りのリストを指定します。

デフォルトの JSON フィールド名の完全なリストについては、 JSON ログ・イベントの参照リストを参照してください。

詳細については、設定可能なJSONログフィールド名を参照してください。

カスタム・メッセージおよびトレース JSON フィールド

JSON 形式のメッセージおよびトレース出力にカスタム・フィールドを追加して、特定の問題またはインシデントに関する情報を収集することができます。 例えば、特定のユーザーからの要求を確認する場合は、そのユーザーの ID でアプリケーション・ログをフィルタリングするためのカスタム・フィールドを追加できます。 セッション ID に別のフィールドを追加して、特定のセッションのアプリケーション・ログを分析およびフィルタリングすることができます。

Open Liberty LogRecordContext API は、ログ・レコードおよびトレース・レコードにカスタム・フィールドを追加できます。 この API は、拡張の値を指定してフィールドを追加します。 LogRecordContext API を使用するには、まず com.ibm.websphere.logging.hpel.LogRecordContext クラスをインポートします。 以下の例は、JSON ログにさまざまな種類のカスタム・フィールドを追加する方法を示しています。

アプリケーション・ログにストリング値フィールドを追加するには、アプリケーションに以下の行を含めることができます。

LogRecordContext.addExtension("userName","bob");

新しく指定されたフィールドは、 addExtension メソッドを実行するのと同じスレッドで作成されたログ項目およびトレース項目に追加されます。 この例では、 userName というカスタム・フィールドが bob ユーザー ID に追加されます。

boolean、float、int、または long 値を持つフィールドを追加するには、以下の例に示すように、拡張名に接尾部 _bool、 _float、 _int、または _longを含める必要があります。

LogRecordContext.addExtension("extensionName_bool","true");
LogRecordContext.addExtension("extensionName_int","112233");
LogRecordContext.addExtension("extensionName_float","1.2");
LogRecordContext.addExtension("extensionName_long","132");

これらの接尾部を指定して非ストリング値を追加する場合、結果の JSON フィールド値はログ内で引用符で囲まれません。 ログでは、ストリング値の JSON フィールド値のみが引用符で囲まれます。

ログからカスタム・フィールドを削除するには、以下の方法を使用します。

LogRecordContext.removeExtension(extensionName);

拡張を削除した後、同じスレッドで作成された後続のログおよびトレースの JSON 出力には、そのフィールドは含まれません。

選択可能な HTTP アクセス・ログ JSON フィールド

ログが JSON 形式で、 accessLog ソースが指定されている場合、デフォルトの HTTP アクセス・ログ JSON フィールドを別のフィールド・セットに置き換えることができます。 jsonAccessLogFields 属性を使用して、アクセス・ログでデフォルト・フィールドを使用するか、 logFormat 属性で指定された一連のフィールドを使用するかを指定できます。 置換フィールドは、 httpEndpoint エレメント内の accessLogging エレメントの logFormat 属性で指定します。

この構成では、リモート・ユーザー ID や要求ヘッダーなど、JSON ログでは使用できない情報を受け取ることができます。 これらのログは、Elastic スタックなどのログ分析ツールがサーバーをモニターするために使用できます。 HTTP アクセス・ログのフォーマット・オプションについて詳しくは、 HTTP アクセス・ロギングを参照してください。

以下の例は、デフォルトの HTTP アクセス・ログ・フィールドを、 logFormat 属性で指定されたフィールドに置き換えるための server.xml ファイル内の構成を示しています。 JSON アクセス・ログを受信するには、 HTTP アクセス・ロギング を有効にする必要があります。

<httpEndpoint httpPort="9080" httpsPort="9443" id="defaultHttpEndpoint">
    <accessLogging logFormat='%R{W} %u %{my_cookie}C %s'/>
</httpEndpoint>
<logging jsonAccessLogFields="logFormat"/>

コンテナー・イメージのロギング構成

コンテナー化された環境では、以下の例に示すように、環境変数を使用してメッセージ・ログを無効にし、コンソール出力を JSON としてフォーマットすることができます。

WLP_LOGGING_MESSAGE_FORMAT=json
WLP_LOGGING_MESSAGE_SOURCE=
WLP_LOGGING_CONSOLE_FORMAT=json
WLP_LOGGING_CONSOLE_LOGLEVEL=info
WLP_LOGGING_CONSOLE_SOURCE=message,trace,accessLog,ffdc,audit

-e フラグを使用して環境変数を設定することにより、 docker run コマンドを実行するときにこの構成を指定できます。

docker run -e "WLP_LOGGING_CONSOLE_SOURCE=message,trace,accessLog,ffdc"
           -e "WLP_LOGGING_CONSOLE_FORMAT=json"
           -e "WLP_LOGGING_CONSOLE_LOGLEVEL=info"
           -e "WLP_LOGGING_MESSAGE_FORMAT=json"
           -e "WLP_LOGGING_MESSAGE_SOURCE=" open-liberty

Podman を使用してコンテナーを管理する場合は、以下のコマンドを実行します。

podman run -e "WLP_LOGGING_CONSOLE_SOURCE=message,trace,accessLog,ffdc"
           -e "WLP_LOGGING_CONSOLE_FORMAT=json"
           -e "WLP_LOGGING_CONSOLE_LOGLEVEL=info"
           -e "WLP_LOGGING_MESSAGE_FORMAT=json"
           -e "WLP_LOGGING_MESSAGE_SOURCE=" open-liberty

バイナリー・ロギング

Liberty には、トレース・ファイルの書き込みに必要なリソースを削減するハイパフォーマンス・バイナリー・ログ・フォーマット・オプションがあります。 最良のパフォーマンスを得るには、バイナリー・ロギングを構成するときに、コンソール・ログ・レベルを OFF に設定し、 代わりにメッセージ・ログを使用してください。 以下の例に示すように、 bootstrap.properties ファイルでバイナリー・ロギングを構成できます。

websphere.log.provider=binaryLogging-1.0
com.ibm.ws.logging.console.log.level=OFF
com.ibm.ws.logging.copy.system.streams=false

ただし、コンソール・ログ・レベルを OFFに設定すると、基礎となる JVM からのメッセージや、ネイティブ標準出力および標準エラー・ストリームに書き込まれるその他のメッセージは、引き続き console.log ファイルに書き込まれます。

以下の defaultServerというサーバーの例に示すように、 binaryLog コマンドを使用して、バイナリー・ログをテキスト・ファイルに変換できます。

binaryLog view defaultServer

時間ベースのログ・ロールオーバー

ログ・ロールオーバーの開始時刻とログ・ロールオーバー間隔の期間を指定することにより、Liberty メッセージおよびトレース・ログの時間ベースの定期的なロールオーバーを有効にすることができます。 指定されたロールオーバー開始時刻は、ログが最初にロールオーバーした時刻です。 ロールオーバー間隔期間は、連続するログ・ロールオーバー間の時間間隔です。

例えば、ロールオーバー開始時刻が午前 0 時で、ロールオーバー間隔が 1 日のサーバーは、メッセージおよびトレース・ログを毎日午前 0 時にロールオーバーします。

Open Liberty メッセージおよびトレース・ログのロールオーバーを構成するには、 bootstrap.properties ファイルを編集するか、環境変数を指定するか、サーバー構成属性を使用します。 以下の例は、これらの各オプションの構成を示しています。ログ・ロールオーバーの開始時刻は午前 0 時、ログ・ロールオーバー間隔は 1 日です。

  • 以下の例では、 bootstrap.properties ファイルで時間ベースのログ・ロールオーバーを構成します。

    com.ibm.ws.logging.rollover.start.time=00:00
com.ibm.ws.logging.rollover.interval=1m

  • 以下の例では、 server.env ファイル内の環境変数を使用して時間ベースのログ・ロールオーバーを構成します。

    WLP_LOGGING_ROLLOVER_START_TIME=00:00
WLP_LOGGING_ROLLOVER_INTERVAL=1m

  • 以下の例では、 server.xml ファイル内のサーバー構成属性を使用して、時間ベースのログ・ロールオーバーを構成します。

    <logging rolloverStartTime="00:00" rolloverInterval="1d"/>

component = level

このコンポーネントは、レベルが設定されているログ・ソースを指定します。 コンポーネントは、ロガー名、トレース・グループ、またはクラス名にすることができます。 このレベルは、以下のいずれかのレベルを使用して、そのコンポーネントについて出力するトレースのレベルを指定します。

off、 fatal、 severe、 warning、 audit、 info、 config、 detail、 fine、 finer、 finest、 all。 トレース・コンポーネントを off、 fatal、 severe、 warning、 audit、または info レベルに設定すると、それを info レベルに設定した場合と同じ効果があり、 info 以上のレベルを無効にすることはできません。

コロンで区切った複数のログ詳細レベルの指定を指定できます。

アスタリスク * は、接頭部に基づいて複数のコンポーネントを突き合わせるためのワイルドカードとして機能します。 例：

- * 製品システムコードと顧客コードを含む、アプリケーションサーバーで実行されているすべての追跡可能なコードを指定します。

- com.ibm.ws.* com.ibm.wsで始まるパッケージ名を持つすべてのクラスを指定します。

- com.ibm.ws.classloading.AppClassLoader AppClassLoaderクラスのみを指定します。

trace.log ファイルが作成され、1 つ以上のトレース・コンポーネントが config レベル以下に設定されている場合にのみ、そのファイルにログが書き込まれます。 ロギング・レベルについて詳しくは、 トレース・ログの有効なログ詳細レベルを参照してください。

com.ibm.ws.logging.newLogsOnStart

Open Liberty の開始時にこの設定が true に設定されている場合、既存の messages.log ファイルまたは trace.log ファイルはロールオーバーされ、ロギングは新しい messages.log ファイルまたは trace.log ファイルに書き込まれます。 この設定が false に設定されている場合、 messages.log ファイルまたは trace.log ファイルは、 maxFileSize 属性で指定されたサイズに達したときにのみリフレッシュされます。 デフォルト値はtrueです。 この設定は、サーバー・ブートストラップ時にのみ処理されるため、 server.xml ファイル内の logging エレメントを使用して指定することはできません。

hideMessage

com.ibm.ws.logging.hideMessage

この設定を使って、console.log と messages.log ファイルから非表示にしたいメッセージキーを設定できます。 メッセージが非表示になると、 trace.log ファイルにリダイレクトされます。

isoDateFormat

com.ibm.ws.logging.isoDateFormat

この設定は、ログ・ファイルで ISO-8601 形式の日付を使用するかどうかを指定します。 デフォルト値は false です。

この設定をtrueにすると、ISO-8601 フォーマットがmessages.logファイル、trace.logファイル、FFDCログで使われる。 形式はyyyy-MM-dd’T’HH:mm:ss.SSSZです。

値 false を指定した場合、日時は、システムで設定されているデフォルト・ロケールに従った形式になります。 デフォルトのロケールが見つからない場合は、dd/MMM/yyyy HH:mm:ss:SSS zのようになります。

maxFfdcAge

com.ibm.ws.logging.max.ffdc.age

WLP_LOGGING_MAX_FFDC_AGE

この設定を使用して、経過時間に基づく FFDC ファイルのパージを有効にします。 この設定は、FFDC ファイルが削除されるまでの最大ファイル存続期間を指定します。 パージは毎日深夜に行われます。 正整数の後に時間単位を指定します。日 (d)、時間 (h)、または分 (m) を指定できます。 例えば、2 日を 2dと指定します。 1 つのエントリーに複数の値を含めることができます。 例えば、2d6hは2日と6時間に相当する。

appsWriteJson

com.ibm.ws.logging.apps.write.json

WLP_LOGGING_APPS_WRITE_JSON

メッセージ・ログやコンソールがJSONフォーマットである場合、この設定により、アプリケーションはJSONフォーマットのメッセージを変更せずに、それらの宛先に書き込むことができます。

jsonAccessLogFields

com.ibm.ws.json.access.log.fields

Wlp_logging_json_access_log_fields

ログが JSON 形式の場合、この設定を使用して、デフォルトの HTTP アクセス・ログ JSON フィールドを、 accesLogging エレメントの logFormat 属性で指定されたフィールドに置き換えることができます。

jsonFieldMappings

com.ibm.ws.logging.json.field.mappings

WLP_LOGGING_JSON_FIELD_MAPPINGS

ログがJSON形式の場合、この設定を使用して、デフォルトのフィールド名を新しいフィールド名に置き換えたり、ログからフィールドを省略したりします。 詳細については、設定可能なJSONフィールド名を参照のこと

logDirectory

com.ibm.ws.logging.log.directory

ログ・ディレクトリー

この設定を使って、console.logファイルを除く、FFDCを含むすべてのログファイルのディレクトリを設定することができます。 デフォルトは WLP_OUTPUT_DIR/serverName/logs です。 server.xml ファイルに logDirectory を設定することは推奨されません。これは、 server.xml ファイルが読み取られる前に、一部のログ・データがデフォルトの場所に書き込まれる可能性があるためです。

maxFiles

com.ibm.ws.logging.max.files

この設定は、保持する各ログ・ファイルの数を指定します。 この設定は、FFDC の例外要約ログの数にも適用されます。 ですから、この数が10の場合、ffdc/ディレクトリに10個のメッセージログ、10個のトレースログ、10個の例外サマリーがあることになります。 デフォルトでは、この値は 2 です。 コンソール・ログはロールしないため、この設定は console.log ファイルには適用されません。

maxFileSize

com.ibm.ws.logging.max.file.size

この設定は、ログファイルがロールバックされるまでの最大サイズ（MB単位）を指定します。 0に設定すると、ログローリングが無効になります。 デフォルト値は 20です。 console.log はロールしないため、この設定は適用されません。

stackTraceSingleEntry

com.ibm.ws.logging.stackTraceSingleEntry

WLP_LOGGING_STACK_TRACE_SINGLE_ENTRY

この属性を trueに設定すると、標準出力および標準エラーに対して発行されたスタック・トレースが単一のログ・イベントにマージされます。 デフォルトでは、この値は false に設定され、スタック・トレースの個々の行が個別のログ・イベントとして発行されます。

rolloverStartTime

com.ibm.ws.logging.rollover.start.time

WLP_LOGGING_ROLLOVER_START_TIME (LP_LOGGING_ROLLOVER_START_TIME)

messages.log ファイルおよび trace.log ファイルの時間ベースのログ・ロールオーバーを有効にするには、この設定を単独で使用するか、 rolloverInterval 属性を指定して使用します。 この設定は、ログが最初にロールオーバーされる予定時刻を指定します。 rolloverInterval 設定の期間は、 rolloverStartTime 属性で指定された時刻に開始します。 有効な値は24時間のISO-8601日付-時刻フォーマットHH:MMに従う。 ゼロを埋め込む必要があります。 rolloverInterval属性が指定されている場合、rolloverStartTime属性のデフォルト値は午前0時00分です。 console.log はロールしないため、この設定は適用されません。

rolloverInterval

com.ibm.ws.logging.rollover.interval

WLP_LOGGING_ROLLOVER_INTERVAL

messages.log ファイルおよび trace.log ファイルの時間ベースのログ・ロールオーバーを有効にするには、この設定を単独で使用するか、 rolloverStartTime 属性を指定して使用します。 この設定では、ログのロールオーバー間の時間間隔を指定します（時間の単位が指定されていない場合は分単位）。 正整数の後に時間単位 (日 (d)、時間 (h)、または分 (m)) を付けて指定してください。 例えば、5 時間を 5hと指定します。 1 つのエントリーに複数の値を含めることができます。 例えば、1d5h は 1 日と 5 時間に相当します。 rolloverStartTime属性が指定されている場合、rolloverInterval属性のデフォルト値は1日です。 console.log はロールしないため、この設定は適用されません。

トレース・ログ詳細レベル

server.xml ファイル内の logging エレメントに traceSpecification 属性を指定して、トレースを選択的に有効にし、サーバー構成内の特定のコンポーネントのトレース・ログ詳細レベルを設定することができます。 以下の例では、2 つのコンポーネント・グループのトレース仕様を指定しています。

<logging traceSpecification="com.myco.mypackage1.*=fine:com.myco.mypackage2.*=finest"/>

この例では、2 つの詳細レベルの指定はコロン (:) で区切られています。

com.myco.mypackage1.*=fine 値は、 com.myco.mypackage1で始まるパッケージ名を持つすべてのクラスに適用されます。

com.myco.mypackage2.*=finest 値は、 com.myco.mypackage2で始まるパッケージ名を持つすべてのクラスに適用されます。

以下の表に、 traceSpecification 属性で指定できる有効なログ詳細レベルをリストします。 任意のコンポーネントに空の値を指定して、そのコンポーネントのすべてのトレースを無効にすることができます。 指定されていないコンポーネントは、デフォルトの状態*=infoに初期化されます。

トレース・コンポーネントを off、 fatal、 severe、 warning、 audit、または info レベルに設定すると、それを info レベルに設定した場合と同じ効果があり、 info 以上のレベルを無効にすることはできません。 trace.log ファイルが作成され、1 つ以上のトレース・コンポーネントが config レベル以下に設定されている場合にのみ、そのファイルにログが書き込まれます。
