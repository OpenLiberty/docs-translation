Server configuration overview

Open Liberty サーバー構成は、1 つの必須ファイル、 server.xml ファイル、および一連のオプション・ファイルで構成されています。 server.xml ファイルは整形式 XML でなければならず、ルート・エレメントは serverでなければなりません。 server.xml ファイルが処理されるときに、認識されないエレメントまたは属性はすべて無視されます。

このサンプル server.xml ファイルは、以下のことを行うようにサーバーを構成します。

<server description="new server">
    <featureManager>
        <feature>jsp-2.3</feature>
    </featureManager>
    <httpEndpoint id="defaultHttpEndpoint"
                  httpPort="9080"
                  httpsPort="9443" />
    <applicationManager autoExpand="true" />
</server>

  • feature エレメントは、 JavaServer Pages 2.3 (jsp-2.3) フィーチャーをサポートするように Open Liberty を構成します。

  • httpPort 属性は 9080に設定されます。これにより、ポート 9080で localhost への着信トラフィックを listen するように Open Liberty が構成されます。

  • 「 autoExpand 」属性は「 true 」に設定されているため、WAR ファイルはデプロイ時に自動的に展開されます。

サーバー構成 という用語は、サーバー構成を構成するすべてのファイル、または XML ファイル内にある構成を指すために使用できます。 コンテキスト内で明確でない場合は、 サーバー XML 構成 という用語を使用して、XML ファイル内の構成を参照することができます。

構成ファイル

サーバー構成ファイルは、以下の順序で処理されます。

  1. server.env -環境変数はこのファイルで指定されます。

  2. jvm.options -JVM オプションはこのファイルで設定されます。

  3. bootstrap.properties -このファイルは、 Open Liberty サーバーの始動に影響を与えます。

  4. server.xml -この必須ファイルは、サーバー構成とフィーチャーを指定します。

server.env

server.env ファイルはオプションです。 これらのファイルは bin/server シェル・スクリプトによって読み取られ、 bin/server スクリプトの動作に影響を与えるために主に使用される環境変数を指定します。 server.env ファイルは、以下の場所から順番に読み取られます。

  1. ${wlp.install.dir}/etc/

  2. ${wlp.user.dir}/shared/

  3. ${server.config.dir}/

同じプロパティーが複数の場所に設定されている場合は、最後に検出された値が使用されます。 シェル環境でこれらの環境変数を指定することもできますが、server.env ファイルが環境変数よりも優先されます。

これらのファイルの最も一般的な使用法は、以下の環境変数を設定することです。

  • JAVA_HOME -使用する JVM を示します。 JVM が設定されていない場合は、システム・デフォルトが使用されます。

  • WLP_USER_DIR -サーバー構成が含まれている usr ディレクトリーの場所を示します。 この環境変数は、 etc/server.env ファイルでのみ設定してください。他の場所は usr ディレクトリーを基準にしているためです。

  • WLP_OUTPUT_DIR -サーバーがファイルを書き込む場所を示します。 デフォルトでは、サーバーは、構成の読み取り元のディレクトリー構造に書き込みます。 ただし、一部のセキュア・プロファイルでは、サーバー構成が読み取り専用であるため、サーバーは別の場所にファイルを書き込む必要があります。

server.env ファイルは、以下の例に示すように、 KEY=value 形式になります。

JAVA_HOME=/opt/ibm/java
WLP_USER_DIR=/opt/wlp-usr

server.env ファイルには、以下の制限が適用されます。

  • キー値にスペースを含めることはできません。

  • 空の行は無視されます。

  • 番号記号 (#) で開始する行 は無視されます。ただし、 Linuxで変数拡張を有効にする # enable_variable_expansion 行は除きます。

  • 等号 (=) を囲む空白文字がないことを確認してください。

  • 拡張変数が使用可能になっている場合を除き、すべての文字はリテラルです。

詳しくは、 デフォルトの環境変数を参照してください。

Expansion 変数

拡張変数は、その値で置き換えられる環境変数またはシェル変数です。 拡張変数は、他の変数の値定義で指定できます。 これらの拡張変数が提供する値は、サーバー構成では環境変数として、シェル環境ではシェル変数として定義できます。 脱字記号 (^) これは、Windows で感嘆符 (!) のような変数拡張文字をエスケープするために使用されます。 拡張変数を指定するための構文は、オペレーティング・システムによって異なります。

変数拡張サポートは、Windows ではデフォルトで有効になっています。 拡張変数を指定するには、Windows の遅延拡張構文 !variable_name!を使用します。これは、変数の値で置き換えられます。

以下の server.env ファイルの例は、Windows での展開変数の構文を示しています。

IBM_DIR=!ProgramFiles!\IBM
JAVA_HOME=!IBM_DIR!\java
WLP_USER_DIR=!USERPROFILE!\wlp-usr

バージョン 22.0.0.1 以降では、 Open Liberty は、 Linux システム上の server.env ファイル内の拡張変数をサポートしています。 拡張変数には、先頭にドル記号 ($) が付きます。 変数名は、オプションで中括弧 ({}) で囲むことができます (例: $variable_name または ${variable_name})。

Linuxの拡張変数を有効にする場合は、単一引用符 (' )、二重引用符 (" )、および円記号 (¥) シェル環境の場合と同じように動作します。 例えば、ストリングを単一引用符で囲むと、変数が展開されなくなります。 ストリングを二重引用符で囲んでも、変数の展開は妨げられません。円記号は、特殊文字の意味を除去するエスケープ文字です。

デフォルトでは、 server.env ファイルは Linuxでの変数拡張をサポートしていません。 各 KEY=value ペアの値は文字どおりに解釈されるため、スペースなどの特殊文字をエスケープする必要はありません。 ただし、変数拡張のサポートを有効にする場合は、特殊文字をエスケープする必要があります。

変数拡張のサポートを有効にするには、 server.env ファイルの先頭に以下のコメントを追加します。

# enable_variable_expansion

拡張変数が使用可能な場合は、スペースを含むすべての値を引用符で囲む必要があります。 内側の引用符もエスケープする必要があります。 例えば、内部引用符は以下の値でエスケープされます。

MESSAGE1='Don\'t do that'

以下の server.env ファイルの例は、 Linuxでの拡張変数の構文を示しています。 WLP_USER_DIR および LOG_DIR 環境変数の値には、拡張変数が含まれています。

# enable_variable_expansion
JAVA_HOME=/opt/ibm/java
WLP_USER_DIR=/home/${USER}/wlp-usr
LOG_DIR=${WLP_USER_DIR}/logs

jvm.options

jvm.options ファイルはオプションです。 これらのファイルは、 Open Liberty用に JVM を起動するときに使用するオプションを決定するために、 bin/server シェル・スクリプトによって読み取られます。 jvm.options ファイルは、以下の場所から順番に読み取られます。

  1. ${wlp.user.dir}/shared/jvm.options

  2. ${server.config.dir}/configDropins/defaults/

  3. ${server.config.dir}/

  4. ${server.config.dir}/configDropins/overrides/

これらの場所に jvm.options ファイルが存在しない場合、サーバー・スクリプトはそのファイルを ${wlp.install.dir}/etcで探します (そのようなディレクトリーが存在する場合)。

jvm.options ファイルの一般的な使用法は、以下のとおりです。

  • JVM メモリー制限の設定

  • モニター製品によって提供される Java エージェントの使用可能化

  • Java システム・プロパティーの設定

jvm.options ファイル・フォーマットでは、以下の例に示すように、JVM オプションごとに 1 行が使用されます。

-Xmx512m
-Dmy.system.prop=This is the value.

スペースなどの特殊文字をエスケープする必要はありません。 オプションが読み取られ、順に JVM に提供されます。 複数のオプションを指定すると、それらはすべて JVM に表示されます。 これらのファイルは、変数置換をサポートしていません。

bootstrap.properties

bootstrap.properties ファイルはオプションです。

このファイルは、 Open Liberty ブートストラップ中に読み取られ、サーバー始動の初期段階の構成を提供します。 これは、 server.xml ファイルより前のサーバーによって読み取られるため、 Open Liberty カーネルの開始時からの始動および動作に影響を与える可能性があります。 bootstrap.properties ファイルは、単純な Java プロパティー・ファイルであり、 ${server.config.dir}にあります。 bootstrap.properties ファイルの一般的な使用法は、 server.xml ファイルが読み取られる前のロギング動作に影響を与える可能性があるため、ロギングを構成することです。

bootstrap.properties ファイルは、特別なオプション・プロパティー bootstrap.includeをサポートしています。このプロパティーは、ブートストラップ・ステージ中にも読み取られる別のプロパティー・ファイルを指定します。 例えば、この boostrap.include ファイルには、複数のサーバーで使用するブートストラップ・プロパティーの共通セットを含めることができます。 bootstrap.include ファイルを絶対ファイル・パスまたは相対ファイル・パスに設定します。

server.xml

最も重要で唯一必要な構成ファイルは、 server.xml ファイルです。 server.xml ファイルは整形式 XML でなければならず、ルート・エレメントは serverでなければなりません。 サーバーでサポートされる正確なエレメントは、構成されている機能によって異なり、不明な構成は無視されます。

Open Liberty は、例外による構成の原則を使用します。これにより、簡潔な構成ファイルが可能になります。 ランタイム環境は、組み込みの構成デフォルト設定のセットから作動します。 これらのデフォルト設定をオーバーライドする構成のみを指定します。

サーバー構成ファイルは、以下の場所から順番に読み取られます。

  • ${server.config.dir}/configDropins/defaults/

  • ${server.config.dir}/server.xml

  • ${server.config.dir}/configDropins/overrides/

${server.config.dir}/server.xml ファイルが存在する必要がありますが、その他のファイルはオプションです。

サーバー形式の XML ファイルをディレクトリーにドロップすることで、構成を柔軟に構成できます。 ファイルは、2 つの configDropins ディレクトリーのそれぞれで、アルファベット順に読み取られます。

configDropins ディレクトリーは、 server.xml ファイルを直接変更することなく、サーバー構成を追加または変更するための便利な方法を提供します。 usr/servers/server_nameの下にあり、以下の 2 つのタイプのサブディレクトリーが含まれています。

  • Overrides (usr/servers/server_name/configDropins/overrides): このディレクトリーは、 server.xml ファイル内の構成をオーバーライドするために使用されます。 例えば、 server.xmlで定義されているポート構成を変更するには、関連する構成ファイルをここに配置します。

  • デフォルト (usr/servers/server_name/configDropins/defaults): このディレクトリーは、 server.xmlで指定されていない構成エレメントのデフォルトを設定するためのものです。 これにより、 server.xml は、このディレクトリーを介して提供される他のデフォルト設定を使用して、1 次構成ソースのままにすることができます。

configDropins/overrides ディレクトリーまたは configDropins/defaults ディレクトリーに配置された構成ファイルに対する変更は、動的にモニターされます。 追加、削除、または更新は、ランタイム構成に自動的に反映されます。

configDropins/overrides ディレクトリー内の構成は、 server.xml ファイル内の構成よりも優先され、 configDropins/defaults ディレクトリー内の構成よりも優先されます。 configDropins/overrides および configDropins/defaults の構成は、フィーチャーが指定するデフォルト設定に優先します。

異なるプラットフォーム間で整合性を維持するために、ファイル名はまず小文字に変換されてからソートされます。 2 つのファイルが同じ名前を共有しているが、大/小文字が異なるシナリオ ( extraConfig.xml や ExtraConfig.xmlなど) では、処理の順序が不整合になる可能性があります。

構成変数の構文

変数は、実行時に実際の値で置換される再使用可能な値を定義します。 Open Liberty サーバー構成ファイルでは、 ${variable.name} 構文を使用して変数を参照できます。 ドル記号 ($) は、変数参照の開始を示します。 中括弧 ({}) は、変数の名前を囲みます。

例えば、 database.url という名前の変数を jdbc:mysql://localhost:3306/mydbという値で定義した場合、 ${database.url}を使用して、その変数をサーバー構成ファイルで参照することができます。 実行時に、 Open Liberty は ${database.url} を実際の値 jdbc:mysql://localhost:3306/mydbに置き換えます。

変数の指定について詳しくは、 変数置換の優先順位を参照してください。

変数置換の優先順位

変数を使用して、サーバー構成をパラメーター化することができます。 値への変数参照を解決するために、以下のソースが優先順位の高い順に参照されます。これは、後のソースが前のソースよりも優先され、優先順位が高いことを意味します。

  1. server.xml ファイル内の変数のデフォルト値

  2. 環境変数

  3. bootstrap.properties

  4. Java システム・プロパティー

  5. ${server.config.dir}/variables ディレクトリー内のファイルからロードされた変数、または VARIABLE_SOURCE_DIRS 環境変数で指定された他のディレクトリーからロードされた変数

  6. server.xml ファイルで宣言された変数値

  7. コマンド行で宣言された変数

変数は、 ${variableName} 構文を使用して参照されます。 以下の例に示すように、サーバー構成で変数を指定します。

<variable name="variableName" value="some.value" />

server.xml ファイルに指定されているデフォルト値は、他の値が指定されていない場合にのみ使用されます。

<variable name="variableName" defaultValue="some.default.value" />

また、始動時にコマンド行から変数を指定することもできます。 これを行うと、コマンド行で指定された変数は、他のすべての変数ソースをオーバーライドするため、サーバーの始動後に変更することはできません。

server run myserver -- --variableName=variableValue

環境変数は変数としてアクセスできます。 環境変数名は直接参照できます。 指定されたとおりに変数を解決できない場合、 server.xml ファイルは、環境変数名の以下のバリエーションを探します。

  • すべての非英数字を下線文字 (_) で置き換えます。

  • すべての文字を大文字に変更

例えば、 server.xml ファイルに ${my.env.var} と入力すると、以下の名前の環境変数が検索されます。

  1. my.env.var

  2. my_env_var

  3. MY_ENV_VAR (V)

VARIABLE_SOURCE_DIRS 環境変数は、変数のロード元となるディレクトリーを指定します。 この環境変数が指定されていない場合、変数ディレクトリーのデフォルトの場所は ${server.config.dir}/variablesです。 この変数の値としてディレクトリーのリストを定義するには、オペレーティング・システムのパス分離文字で各ディレクトリーを区切ります。 Windows の場合、パス分離文字はセミコロン (;) です。 UNIX ベースのシステムの場合、パス分離文字はコロン (:) です。

ファイルの名前が変数名として使用され、ファイルの内容が変数値として使用されます。 例えば、 httpPort という名前の変数ディレクトリーに、唯一の内容としてテキスト・ストリング 9080 を持つファイルを作成できます。 その後、 server.xml ファイルで変数 ${httpPort} を使用すると、 9080に解決されます。

変数ディレクトリー内のディレクトリーにファイルを作成すると、親ディレクトリーが変数名に追加されます。 例えば、変数ディレクトリー内の ports サブディレクトリーに ${httpPort} ファイルを作成することができます。変数名は ${ports/httpPort}です。

VARIABLE_SOURCE_DIRS ディレクトリー内の変数は、プロパティー・ファイルでも定義できます。 ファイルの名前が .propertiesで終わる場合、ファイル内の各プロパティーを使用して変数が作成されます。 例えば、以下の内容を持つ ports.properties という名前のファイルを作成できます。

httpPort=9080
httpsPort=9443

この構成では、 server.xml ファイルに ${httpPort} 変数と ${httpsPort} 変数を指定することができ、それらの値は 9080 と 9443に解決されます。

変数値は、常に単純型変換のストリングとして解釈されます。 したがって、ポートのリスト ( 80,443など) は、2 つのポート番号としてではなく、単一のストリングとして解釈される可能性があります。 以下の例に示すように、 list 関数を使用して、 , で変数置換を強制的に分割することができます。

<mongo ports="${list(mongoPorts)}" hosts="${list(mongoHosts)}" />

整数値を持つ変数に対しては、単純な算術計算がサポートされます。 演算子の左側と右側は、変数または数値のいずれかにすることができます。 演算子は、以下の例に示すように、 +、 -、 *、または /にすることができます。

<variable name="one" value="1" />
<variable name="two" value="${one+1}" />
<variable name="three" value="${one+two}" />
<variable name="six" value="${two*three}" />
<variable name="five" value="${six-one}" />
<variable name="threeagain" value="${six/two}" />

以下の事前定義変数が存在します。

  • wlp.install.dir - Open Liberty ランタイムがインストールされているディレクトリー。

  • wlp.server.name は、サーバーの名前です。

  • wlp.user.dir - usr フォルダーのディレクトリーです。 デフォルトは${wlp.install.dir}/usrです。

  • shared.app.dir -共有アプリケーションのディレクトリー。 デフォルトは ${wlp.user.dir}/shared/apps です。

  • shared.config.dir -共有構成ファイルのディレクトリー。 デフォルトは ${wlp.user.dir}/shared/config です。

  • shared.resource.dir -共有リソース・ファイルのディレクトリー。 デフォルトは ${wlp.user.dir}/shared/resources です。

  • server.config.dir -サーバー構成が保管されているディレクトリー。 デフォルトは ${wlp.user.dir}/servers/${wlp.server.name} です。

  • server.output.dir -サーバーが作業域ファイル、ログ・ファイル、およびその他のランタイム生成ファイルを書き込むディレクトリー。 デフォルトは ${server.config.dir} です。

構成のマージ

構成は複数のファイルで構成できるため、2 つのファイルが同じ構成を提供する可能性があります。 このような状況では、一連の単純なルールに従ってサーバー構成がマージされます。 Open Libertyでは、構成はシングルトン構成とファクトリー構成に分離され、それぞれにマージに関する独自の規則があります。 シングルトン構成は、単一のエレメント (例えば、ロギング) を構成するために使用されます。 ファクトリー構成は、アプリケーション全体やデータ・ソースなどの複数のエンティティーを構成するために使用されます。

シングルトン構成をマージしています

複数回指定されたシングルトン構成エレメントの場合、構成はマージされます。 異なる属性を持つ 2 つのエレメントが存在する場合は、両方の属性が使用されます。 例:

<server>
    <logging a="true" />
    <logging b="false" />
</server>

次のように扱われます。

<server>
    <logging a="true" b="false" />
</server>

同じ属性を 2 回指定すると、最後のインスタンスが優先されます。 例:

<server>
    <logging a="true" b="true" />
    <logging b="false" />
</server>

次のように扱われます。

<server>
    <logging a="true" b="false" />
</server>

構成は、テキストを取る子エレメントを使用して提供されることがあります。

これらのケースでは、指定されたすべての値を使用して構成がマージされます。 最も一般的なシナリオは、フィーチャーの構成です。 例:

<server>
    <featureManager>
        <feature>servlet-4.0</feature>
    </featureManager>
    <featureManager>
        <feature>restConnector-2.0</feature>
    </featureManager>
</server>

次のように扱われます。

<server>
    <featureManager>
        <feature>servlet-4.0</feature>
        <feature>restConnector-2.0</feature>
    </featureManager>
</server>

ファクトリー構成のマージ

ファクトリー構成マージでは、singleton 構成と同じ規則が使用されますが、要素名が一致するだけでは要素が自動的にマージされない点が異なります。 ファクトリー構成では、同じエレメントを構成することができ、2 つの異なる論理オブジェクトを意味します。 したがって、各エレメントは別個のオブジェクトを構成するものと想定されます。 1 つの論理オブジェクトが 2 つのエレメントによって構成されている場合、それらが同じものであることを示すために、各エレメントに id 属性を設定する必要があります。 id 属性での変数置換はサポートされていません。

以下の例では、2 つのアプリケーションを構成します。 最初のアプリケーションは myapp.warで、コンテキスト・ルートは myawesomeappです。 もう 1 つのアプリケーションは myapp2.warで、これにはコンテキスト・ルートとして myapp2 があります。

<server>
    <webApplication id="app1" location="myapp.war" />
    <webApplication location="myapp2.war" />
    <webApplication id="app1" contextRoot="/myawesomeapp" />
</server>

処理を組み込む ( I)

include エレメントを使用して、サーバー構成に追加の構成リソースを組み込むことができます。 サーバー構成ファイルに別のリソースへの組み込み参照が含まれている場合、サーバーは、参照されているリソースの内容を、 include エレメントの代わりにインラインで組み込まれているかのように処理します。

以下の例では、サーバーは、 other2.xml ファイルの内容を処理する前に、 other.xml ファイルの内容を処理します。

<server>
    <include location="other.xml" />
    <include location="other2.xml" />
</server>

include エレメントの location 属性は、リモート・リソースの相対ファイル・パスまたは絶対ファイル・パス、ディレクトリー、あるいは URI に設定できます。

ディレクトリーを location 値として指定すると、サーバーは指定されたディレクトリー内にあるファイルのみを処理し、サブディレクトリーは無視します。 サーバーは、ディレクトリー内のファイルをアルファベット順に処理します。

<server>
    <include location="./common/"/>
</server>

この例では、 ./common/ ディレクトリー内の構成ファイルは、サーバーの始動時にアルファベット順に処理されます。

サーバーは、ディレクトリー内の構成ファイルを処理するときに、ファイル名を小文字に切り替えてアルファベット順にソートし、ファイル処理順序の一貫性と予測可能性を確保します。 ただし、大/小文字のバリエーション ( extraConfig.xml や ExtraConfig.xmlなど) を除き、同じディレクトリーに同じ名前で 2 つのファイルが指定されている場合、順序付けの動作は不確定になります。

include の場所にディレクトリーを指定する場合は、ディレクトリー名にスラッシュ (/) を付加する必要があります。 例えば、location="./common/" などです。 ディレクトリー名 (location="./common") の後にスラッシュを省略すると、エラーになります。

リモート・リソースの URI から構成を組み込む場合、 location 属性は、ディレクトリーではなく、単一のリモート・リソース・ファイルを指す必要があります。 例えば、 https://website.com/path/to/server.xml はサポートされていますが、 https://website.com/path/to/directory はサポートされていません。

デフォルトでは、インクルード・リソースが存在している必要があります。 リソースが使用できない可能性がある場合は、以下の例に示すように、 optional 属性を trueに設定します。

<server>
    <include location="other.xml" optional="true" />
</server>

リソースを組み込むときに、 onConflict 属性を指定して通常のマージ規則を変更することができます。 onConflict 属性の値を IGNORE または REPLACE 競合する任意の構成に設定できます。

<server>
    <include location="other.xml" onConflict="IGNORE" />
    <include location="other2.xml" onConflict="REPLACE" />
</server>

include エレメントを使用してディレクトリーから取り込まれる構成ファイルの場合、 onConflict 属性はそのディレクトリー内の各ファイルに個別に適用されます。 例:

<server>
    <include location="/directory/" onConflict="REPLACE" />
</server>

機能的には、この構成は、以下の例に示すように、各ファイルをそれぞれ独自の onConflict 属性で個別に組み込むことと同じです。

<server>
    <include location="a.xml" onConflict="REPLACE" />
    <include location="b.xml" onConflict="REPLACE" />
</server>

構成参照

Open Liberty のほとんどの構成は自己完結型ですが、多くの場合、構成を共有すると便利です。 例えば、 JDBC ドライバー構成を複数のデータ・ソースで共有することができます。 server エレメントの直接の子として定義されているファクトリー構成エレメントを参照できます。

構成への参照では、参照されているエレメントの id 属性が常に使用されます。 参照を行う構成エレメントは、以下の例に示すように、常に Refで終わる属性を使用します。

<server>
  <dataSource jndiName="jdbc/fred" jdbcDriverRef="myDriver" />
  <jdbcDriver id="myDriver" />
</server>

動的更新

サーバーは、更新についてサーバー XML 構成をモニターし、変更が検出されると動的に再ロードします。 非 XML ファイル (server.env、 bootstrap.properties、および jvm.options) に対する変更は、始動時にのみ読み取られるため、動的ではありません。 ローカル・ディスク上のサーバー XML 構成ファイルは、 500msごとに更新がモニターされます。 XML 構成ファイル・モニターの頻度を構成できます。 例えば、10 分ごとにモニターするようにサーバーを構成するには、次のように指定します。

<config monitorInterval="10m" />

MBean が通知されたときにのみファイル・システムのポーリングおよび再ロードを使用不可にするには、次のように指定します。

<config updateTrigger="mbean" />

ログ・メッセージ

サーバーが実行されると、構成を参照するログ・メッセージが出力される場合があります。 ログ内の参照は、XPath に似た構造を使用して構成エレメントを指定します。 エレメント名は、大括弧で囲まれた id 属性の値で指定されます。 サーバー構成に id が指定されていない場合、 id が自動的に生成されます。 以下のサーバー XML 構成の例に基づいて、 dataStore エレメントおよび子 dataSource は、ログ内で dataStore[myDS] および dataStore[myDS]/dataSource[default-0]として識別されます。

<server>
    <dataStore id="myDS">
        <dataSource />
    </dataStore>
</server>
