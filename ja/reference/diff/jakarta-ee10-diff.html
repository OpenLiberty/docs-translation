Jakarta EE 10 と 9.1 の相違点

Jakarta EE 9.1 フィーチャーを使用するアプリケーションを、Jakarta EE 10 フィーチャーを使用するアプリケーションに更新する場合、API の動作の変更により、アプリケーション・コードの更新が必要になることがあります。 これらの変更は、いくつかの Jakarta EE API、およびそれらをサポートする Open Liberty フィーチャーに適用されます。

以下のセクションでは、さまざまな Jakarta EE 10 フィーチャーの更新について説明します。

  • Jakarta EE 10 と 9.1

  • Jakarta Authentication 3.0

  • Jakarta Authorization 2.1

  • Jakarta Batch 2.1

  • Jakarta Contexts and Dependency Injection 4.0

  • Jakarta 式言語 5.0

  • Jakarta Faces 4.0

  • Jakarta メール 2.1

  • Jakarta RESTful Web Services 3.1

  • Jakarta Security 3.0

  • Jakarta Server Pages 3.1

  • Jakarta サーブレット 6.0

  • Jakarta WebSocket 2.1

  • Jakarta XML バインディング 4.0

  • Jakarta XML Web サービス 4.0

Jakarta EE 10 と 9.1

以下の変更は、Jakarta EE 10 プラットフォーム全体に適用されます。

Java SE の要件

Jakarta EE 9.1 フィーチャーは、Java SE バージョン 8 以降で実行できます。 Jakarta EE 10 のすべてのフィーチャーは、Jakarta EE 10 仕様で概説されているように、Java 11 以降を必要とします。

新規コア・プロファイル

Jakarta EE 10 では、Core Profile と呼ばれる新しいプロファイルが導入されています。 コア・プロファイル機能を提供する Liberty フィーチャーは、 Jakarta Contexts and Dependency Injection 4.0 (cdi-4.0)、 Jakarta JSON Binding 3.0 (jsonb-3.0)、 Jakarta JSON Processing 2.1 (jsonp-2.1)、および Jakarta RESTful Web Services 3.1 (restfulWS-3.1) です。 コア・プロファイルには便利な機能は提供されていません。 ただし、 restfulWS-3.1 ( cdi-4.0 と jsonp-2.1の両方を含む) と jsonb-3.0 の両方の Liberty フィーチャーを server.xml ファイルに組み込むことで、すべての Jakarta 10 Core Profile フィーチャーを簡単に有効にすることができます。

XML バインディング API の可用性

いくつかの Jakarta EE 9.1 Liberty フィーチャーには、 Jakarta XML Binding 3.0 (xmlBinding-3.0) フィーチャーへの依存関係が含まれます。これは、これらのフィーチャーが XML バインディング API に依存したり、XML バインディング API を公開したりすることを仕様で要求されていない場合でも同様です。 さらに、これらのフィーチャーの多くは、XML バインディング・フィーチャーを含まないフィーチャーの Jakarta Web Profile サブセットの一部です。

以下の Jakarta EE 10 Liberty フィーチャーは、同等の Jakarta EE 9.1 フィーチャーが XML Binding 3.0 API を公開する場合、XML Binding 4.0 API を公開しません。

  • Jakarta Authentication 3.0 (appAuthentication-3.0)

  • Jakarta Connectors 2.1 (connectors-2.1)

  • Jakarta Persistence 3.1 (persistence-3.1)

  • Jakarta Persistence Container 3.1 (persistenceContainer-3.1)

  • Jakarta EE Web Profile 10.0 (webProfile-10.0)

XML バインディング API を必要とする Open Liberty アプリケーションの場合、 Jakarta XML Binding 4.0 (xmlBinding-4.0) Liberty フィーチャーを server.xml ファイルに追加する必要があります。

管理 REST コネクターの可用性

Jakarta EE Platform 9.1 (jakartaee-9.1) Liberty コンビニエンス・フィーチャーには、 Admin Rest Connector 2.0 (restConnector-2.0) Liberty フィーチャーが含まれています。 Jakarta EE Platform 10.0 (jakartaee-10.0) Liberty コンビニエンス・フィーチャーには、この Liberty フィーチャーは含まれません。

jakartaee-10.0 コンビニエンス・フィーチャーを使用していて、アプリケーションが Admin Rest コネクター機能を必要とする場合は、 restConnector-2.0 フィーチャーを server.xml ファイルに追加する必要があります。

Connectors Inbound Security の置き換え

Jakarta EE 10 では、Jakarta Connectors 2.1 (connectors-2.1) と Application Security 5.0 (appSecurity-5.0) Liberty フィーチャーの組み合わせが、Jakarta EE 9.1で提供される Jakarta Connectors Inbound Security 2.0 (connectorsInboundSecurity-2.0) Liberty フィーチャーに置き換わります。

JSON Web Token (JWT) フィーチャーの変更

JSON Web Token 1.0 (jwt-1.0) Liberty フィーチャーを Jakarta EE 10 Liberty フィーチャーと共に使用可能にすると、サーブレット API はデフォルトで使用できなくなります。 サーブレット API を必要とする Open Liberty アプリケーションの場合、 Jakarta Servlet 6.0 (servlet-6.0) フィーチャーを server.xml ファイルに追加する必要があります。

同様に、 jwt-1.0 フィーチャーが Jakarta EE 10 Liberty フィーチャーで使用可能になっている場合、 com.ibm.wsspi.rest.handler、 com.ibm.wsspi.rest.handler.helper、および com.ibm.wsspi.wab.configure SPI は、ユーザー・フィーチャーで使用できなくなりました。 これらの SPI を必要とする Open Liberty ユーザー・フィーチャーの場合、 com.ibm.websphere.appserver.restHandler-1.0 フィーチャーまたは com.ibm.wsspi.appserver.webBundle-1.0 フィーチャーをユーザー・フィーチャーの mf ファイルに追加する必要があります。

Jakarta Authentication 3.0 と 2.0

Jakarta Authentication 3.0 フィーチャー (appAuthentication-3.0) により、 jakarta.security.auth.message.config.AuthConfigFactory 抽象クラスに対する変更が導入されました。 このクラスは、サーバー認証モジュールを登録および削除するために、以下の 2 つの新しいメソッドを実装します。

  • public abstract String registerServerAuthModule(ServerAuthModule serverAuthModule, Object context);

  • public abstract void removeServerAuthModule(Object context);

この機能の以前のバージョンでは、構成プロバイダーのみを登録および削除するためのサポートが提供されていました。

詳しくは、 Jakarta Authentication 3.0 Javadocを参照してください。

Jakarta Authorization 2.1 と 2.0

Jakarta Authorization 2.1 フィーチャー (appAuthorization-2.1) では、以下の新しいメソッドが導入されています。

jakarta.security.jacc.PolicyConfigurationFactory 抽象クラスの新規メソッド

jakarta.security.jacc.PolicyConfigurationFactory 抽象クラスは、ポリシー構成インターフェース (PolicyConfiguration) を取得するために、以下の 2 つの新規メソッドを実装します。

  • public abstract PolicyConfiguration getPolicyConfiguration();

  • public abstract PolicyConfiguration getPolicyConfiguration(String contextID);

これらのメソッドは、以前のバージョンで使用可能であった以下の getPolicyConfiguration メソッドに加えて使用可能です。

public abstract PolicyConfiguration getPolicyConfiguration(String contextID, boolean remove);

jakarta.security.jacc.PolicyConfiguration インターフェースの新規メソッド

jakarta.security.jacc.PolicyConfiguration インターフェースは、権限を読み取るための 3 つの新規メソッドを定義します。

  • public abstract Map<String, PermissionCollection> getPerRolePermissions();

  • public abstract PermissionCollection getUncheckedPermissions();

  • public abstract PermissionCollection getExcludedPermissions();

詳しくは、 Jakarta Authorization 2.1 Javadocを参照してください。

Jakarta Batch 2.1 と 2.0

Jakarta Batch 2.1 の更新は、主に Context Dependency Injection (CDI) との統合の改善に重点を置いています。 バッチ成果物のディスカバリーが CDI で標準化されるようになりましたが、これは以前の仕様では完全には定義されていませんでした。

@BatchProperty アノテーションの更新

@BatchProperty アノテーションが拡張され、 Boolean、 Short、 Integer、 Long、 Float、および Doubleの各タイプの注入が可能になりました。 以前は、このアノテーションは String 値のみを注入できました。 以下の例は、新しく追加された注入タイプを示しています。

@Inject @BatchProperty(name="b1") Boolean b1;
@Inject @BatchProperty(name="s1") Short s1;
@Inject @BatchProperty(name="i1") Integer i1;
@Inject @BatchProperty(name="l1") Long l1;
@Inject @BatchProperty(name="f1") Float f1;
@Inject @BatchProperty(name="d1") Double d1;

JobOperator インターフェースのデフォルトの注入

以前は、デフォルトの JobOperator インターフェースは、 BatchRuntime.getJobOperator() 静的ファクトリー・メソッドからのみ使用可能になっていました。 Jakarta Batch 2.1 以降では、 JobOperator インターフェースの注入ポイントが存在し、ユーザー提供の実装が見つからない場合、デフォルトの JobOperator インスタンスが自動的に注入されます。 以下の例は、 JobOperator インターフェースのデフォルトの注入を示しています。

@Inject JobOperator jobOperator;

Jakarta Contexts and Dependency Injection 4.0 と 3.0

Jakarta Contexts and Dependency Injection 4.0 フィーチャー (cdi-4.0) は、ビルド互換の拡張および観察可能なコンテナー状態イベントを導入します。 また、空およびバージョン管理されていない beans.xml ファイルの処理方法も変更されます。 このリリースでは、以前に非推奨になった API がいくつか削除されました。

互換性のある拡張機能のビルド

CDI には、CDI アプリケーションの初期化ライフサイクルをカスタマイズするためのポータブル拡張機能が用意されています。 CDI 4.0では、ビルド互換の拡張機能により、ビルド時の処理に合わせて拡張機能を実装することができます。

ビルド互換拡張を実装するには、 META-INF/services ディレクトリーで宣言されている BuildCompatibleExtension インターフェースの実装を提供します。 実装では、以下の拡張アノテーションのいずれかでアノテーションが付けられたメソッドを提供できます。これらのアノテーションはそれぞれ、拡張実行フェーズに対応しています。

  • @Discovery

  • @Enhancement

  • @Registration

  • @Synthesis

  • @Validation

例えば、次の例の実装では、Enhancement フェーズの一部として MyQualifier アノテーションが MyService タイプに追加されます。

public class MyExtension implements BuildCompatibleExtension {
    @Enhancement(type=MyService.class)
    public void addMyQualifier(ClassConfig clazz) {
        clazz.addAnnotation(MyQualifier.class)
    }
}

始動およびシャットダウン・イベント

空および非空の beans.xml ファイル内の bean-discovery-mode 属性

CDI 4.0では、 bean-discovery-mode 属性を指定していない beans.xml ファイルは、属性が bean-discovery-mode="annotated"に設定されているかのように扱われます。 CDI 3.0 以前では、 bean-discovery-mode 属性を指定していない beans.xml ファイルは、属性が bean-discovery-mode="all"に設定されているかのように扱われます。 これらの設定は、空と非空の両方の beans.xml ファイルに適用されます。

CDI 4.0の空の beans.xml ファイルの場合、 cdi エレメントの emptyBeansXmlCDI3Compatibility 属性を設定して、以前のバージョンとの互換性を有効にすることができます。 server.xml ファイルでこの属性を true に設定すると、空の beans.xml ファイルを含むすべてのアーカイブが明示的 Bean アーカイブとして扱われます。 詳しくは、 Server.xml 構成エレメントおよび属性の更新を参照してください。

ベスト・プラクティスは、以下の例に示すように、空でない beans.xml ファイルのバージョンを常に指定することです。

<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xmlns="https://jakarta.ee/xml/ns/jakartaee"
    xsi:schemaLocation="https://jakarta.ee/xml/ns/jakartaee https://jakarta.ee/xml/ns/jakartaee/beans_4_0.xsd"
    version="4.0"
    bean-discovery-mode="annotated">
</beans>

Bean およびインスタンスのプログラマチックな検索

CDI 4.0では、Bean メタデータのプログラマチックな検査を簡素化するために、新しい Handle API を使用できます。 この API を使用すると、必要になる前にインスタンスを作成する必要がなくなります。 Handle インスタンスは、以下の構文を使用する Instance API を使用して取得できます。

public interface Handle<T> extends AutoCloseable {
        T get();
        Bean<T> getBean();
        void destroy();
        void close();
}

Server.xml 構成エレメントおよび属性の更新

Liberty Jakarta Contexts and Dependency Injection 4.0 フィーチャーでは、 cdi12 構成エレメントは、CDI バージョン 1.2 以降に適用される cdi エレメントによって置き換えられます。 以下の server.xml ファイルの例は、2 つの構成属性を持つ cdi エレメントを示しています。

<cdi enableImplicitBeanArchives="false" emptyBeansXmlCDI3Compatibility="true"/>

enableImplicitBeanArchives 属性は、以前のバージョンで cdi12 エレメントを使用した場合と同じように機能します。 この属性がデフォルトの trueに設定されている場合、 beans.xml ファイルを持たないアーカイブは、暗黙的な Bean アーカイブとして扱われ、Bean 定義アノテーションを持つクラスについてスキャンされます。 この属性が falseに設定されている場合、 beans.xml ファイルを含まないアーカイブでは、アノテーション付きクラスはスキャンされません。

emptyBeansXmlCDI3Compatibility 属性は、CDI 4.0にのみ適用されます。 この属性が trueに設定されている場合、空の beans.xml ファイルを含むアーカイブは、CDI 3.0 以前と同様に、明示的 Bean アーカイブとして扱われます。 この属性がデフォルトの falseに設定されている場合、空の beans.xml ファイルを含むアーカイブは暗黙的 Bean アーカイブとして扱われます。

詳しくは、 Jakarta Contexts and Dependency Injection 4.0 Javadoc を参照してください。

Cookie クラスのコンストラクターは、 Cookie.Builder クラスを優先して非推奨になりました。 NewCookie クラスのコンストラクターは、 NewCookie.Builder クラスを優先して非推奨になりました。

Jakarta Security 3.0 と 2.0

Open Libertyでは、 Jakarta Security 3.0 仕様 は、 Application Security 5.0 フィーチャーによってサポートされます。 この機能では、組み込みの OpenID Connect 認証メカニズムを構成するための新しい @OpenIdAuthenticationMechanismDefinition アノテーションが導入されました。 この認証メカニズムは、 OpenID Connect クライアントまたは依拠当事者 (RP) として機能するため、 OpenID Connect プロバイダー (OP) をシングル・サインオン ID プロバイダーとして使用できます。 また、アクセス・トークン、ID トークン、およびリフレッシュ・トークンを処理するための OpenIdContext Bean および各種 API も紹介しています。

詳しくは、 Jakarta Security 3.0 Javadoc および 「 OpenID Connect クライアントをアプリケーション用に使用可能にする」 トピックを参照してください。

Jakarta Server Pages と 3.1 および 3.0

Jakarta Server Pages 3.1 フィーチャーは、 Jakarta Server Pages 3.1 および Jakarta Standard Tag Library 3.0 仕様のサポートを提供します。 このリリースには、式言語の ID およびリゾルバー、新規タグ URI などの更新が含まれています。

不明な ID を検出しています

このリリースでは、不明な式言語 ID を検出するための新しいオプションが追加されました。 このオプションを有効にすると、不明な ID が検出された場合に PropertyNotFoundException が発生します。 以前は、不明な ID は空ストリング ("") としてレンダリングされていました 識別するのは困難でした このオプションは、ページ・ディレクティブを使用するか、タグ・ディレクティブを使用するか、または web.xml ファイル内の JSP プロパティー・グループとして使用可能にすることができます。

ページ・ディレクティブまたはタグ・ディレクティブを使用してこのオプションを有効にするには、以下の例に示すように、 errorOnELNotFound 属性を trueに設定します。

<%@ page errorOnELNotFound="true" %>

web.xml ファイルで JSP プロパティー・グループを使用してこのオプションを有効にするには、以下の例に示すように、 error-on-el-not-found エレメントを trueに設定します。

<jsp-property-group>
   <url-pattern>*.jsp</url-pattern>
   <error-on-el-not-found>true</error-on-el-not-found>
</jsp-property-group>

詳しくは、Jakarta Server Pages 仕様の「 Unknown EL Identifiers 」を参照してください。

新規式言語リゾルバー

インポートおよび未解決の変数の処理を改善するために、式言語リゾルバーが更新されました。 このリリースでは、以下の 2 つの新しいリゾルバーが使用可能です。

ImportELResolver

この新しいインポート・リゾルバーは、 ScopedAttributeELResolver クラスからリファクタリングされ、 import 属性からのクラスまたはパッケージのインポートを式言語環境で使用できるようにします。 以前は、インポートはスクリプト環境でのみ使用可能でした。式がインポート・クラスを使用していた場合は、空ストリングに解決されていました。 詳しくは、 ImportELResolverを参照してください。

NotFoundELResolver

このリゾルバーは、他のリゾルバーが解決しない場合は常に変数を解決します。 デフォルトでは null が返されますが、前述の error-on-el-not-found エレメントを使用してこの動作を変更できます。 詳しくは、 NotFoundELResolverを参照してください。

更新のインポート ( I)

スクリプト環境のデフォルトおよび暗黙のインポートは、式言語環境にも適用されるようになりました。 これらのデフォルトのインポートは、 java.lang.*、 jakarta.servlet.*、 jakarta.servlet.jsp.*、および jakarta.servlet.http.*です。

名前空間の更新

Jakarta Tag URI で jakarta.tags.* パターンが使用されるようになりました。 以前の http://xmlns.jcp.org/jsp/jstl/* パターンのサポートは以前のバージョンとの互換性のために維持されていますが、これらの新しい URI を使用するようにアプリケーションを更新することをお勧めします。 以下の例は、新しい URI を持つ taglib ディレクティブを示しています。

<%@ taglib prefix="c" uri="jakarta.tags.core" %>
<%@ taglib prefix="x" uri="jakarta.tags.xml" %>
<%@ taglib prefix="fmt" uri="jakarta.tags.fmt" %>
<%@ taglib prefix="fn" uri="jakarta.tags.functions" %>"
<%@ taglib prefix="sql" uri="jakarta.tags.sql" %>

詳しくは、 Jakarta Tags docを参照してください。

非推奨

Servlet 6.0 API で SingleThreadModel インターフェースが削除されたため、 isThreadSafe ディレクティブは推奨されません。 回避策が実装されていますが、パフォーマンスに重大な影響が生じる可能性があります。 ページ 3.1 での isThreadSafe ディレクティブの使用は推奨されません。

jsp:plugin、 jsp:params、および jsp:fallback の各アクションは非推奨になっており、作動可能ではありません。 現在のブラウザーでは、 jsp:plugin および関連アクションによって生成される要素がサポートされなくなりました。

Jakarta Servlet 6.0 と 5.0

Jakarta Servlet 6.0 フィーチャー (servlet-6.0) は、応答 Cookie に属性を設定するための新規 API および新規構成を提供します。

Cookie 属性を構成する新しい方法

Jakarta Servlet 6.0では、デプロイメント記述子ファイルから、 jakarta.servlet.ServletContainerInitializer API を使用したアプリケーションの開始時、または要求時に、応答 Cookie に属性を設定することができます。

web.xml デプロイメント記述子ファイルに Cookie 属性を設定するには、以下の例に示すように、 cookie-config エレメント内に attribute エレメントを指定します。

<session-config>
   <cookie-config>
     <name>SampleCookieName</name>
     <domain>my.domain.com</domain>
     <path>/</path>
     <http-only>true</http-only>
     <secure>true</secure>
     <attribute>
        <attribute-name>MyAttributeName</attribute-name>
        <attribute-value>valueExample</attribute-value>
     </attribute>
     <attribute>
        <attribute-name>MyOtherAttribute</attribute-name>
        <attribute-value>OtherValue</attribute-value>
     </attribute>
   </cookie-config>
</session-config>

以下の例に示すように、 jakarta.servlet.ServletContainerInitializer API を使用して、アプリケーション始動時にアプリケーションから Cookie 属性を動的に設定することもできます。

@Override
public void onStartup(Set<Class<?>> setOfClasses, ServletContext context) throws ServletException {
      SessionCookieConfig scc = context.getSessionCookieConfig();
      scc.setName("MyDynaCookieName");
      scc.setAttribute("MyDynamicAttribute", "DynamicValue");
      ...
}

あるいは、以下の例に示すように、要求時にアプリケーションから属性を設定することもできます。

SessionCookieConfig scc = getServletContext().getSessionCookieConfig();
scc.setName("MyRequestCookieName");
scc.setAttribute("MyRequestAttribute", "RequestValue");

Cookie 構成のさまざまなソース間で競合が発生した場合、競合を解決するために、以下の優先順位が適用されます (優先順位の高いものから低いものの順)。

  1. 要求時にアプリケーションによって設定される属性

  2. ServletContainerInitializer API を使用してアプリケーションによって設定される属性

  3. web.xml デプロイメント記述子ファイルで設定される属性

要求 Cookie ヘッダーに対する変更

Servlet 6.0 は、ブラウザーなどの HTTP クライアント・エージェントが Cookie 属性をアプリケーション・サーバーに返送しないことを規定する rfc6265 標準に従って、HTTP 要求 Cookie の動作を実装しています。

以前は、有効な属性名に接頭部としてドル記号 ($) を追加することによって、要求 Cookie 属性を組み込むことができました。 例えば、 $Domain=myDomain.com は、要求 Cookie の Domain 属性として扱われました。 Servlet 6.0以降、 $Domain は、 $Domainという名前の新規 Cookie として扱われます。 ドル記号は Cookie 名の一部でもあります。 この規則の唯一の例外は、 $Version 値です。

例えば、Cookie ヘッダーを含む以下の HTTP 要求について考えてみます。

Cookie: $Version=1; name1=value1; $Path=/Dollar_Path; $Domain=localhost; $NAME2=DollarNameValue;Domain=DomainValue

Servlet 6.0より前では、サーバーはこの要求から以下の 3 つの要求 Cookie を作成していました。

cookie1: name1 with the associated attributes: Path=/Dollar_Path; Domain=localhost
cookie2: NAME2
cookie3: Domain

Servlet 6.0 以降では、サーバーは、同じ HTTP 要求から以下の 5 つの要求 Cookie を作成します。

cookie1: name1
cookie2: $Path
cookie3: $Domain
cookie4: $NAME2
cookie5: Domain

アプリケーション・デバッグの機能拡張

新しい jakarta.servlet.ServletRequest API と新しい jakarta.servlet.ServletConnection クラスを使用して、アプリケーションへの要求をデバッグすることができます。 この機能拡張により、要求を追跡または参照したり、要求 ID を使用して要求で使用されるネットワーク接続の詳細を取得したりすることができます。

Servlet 6.0: では、以下の Jakarta.servlet.ServletRequest API が使用可能です。

  • String getRequestId()

  • String getProtocolRequestId()

  • ServletConnection getServletConnection()

新しい jakarta.servlet.ServletConnection オブジェクトが getServletConnection() サーブレット要求から取得されます。 これには、以下の API が含まれています。

  • String getConnectionId()

  • String getProtocol()

  • String getProtocolConnectionId()

  • boolean isSecure()

URI パスの検証

Servlet 6.0 リリース以降、以下のいずれかのシーケンスが URI に存在する場合、直接要求は 400 Bad Request 状況で拒否されます。

  • エンコードされたドット (%2E) 文字

  • エンコードされたスラッシュ (%2F) 文字

  • フラグメント (#) 文字 (エンコードされているか (%23) されていないかにかかわらず)

  • 円記号 (\) (エンコードされているか (%5C) されていないかにかかわらず)

  • エンコードされているかどうかにかかわらず、任意の制御文字

  • パス・パラメーター (;) を持つ任意のピリオド (.) または二重ピリオド (..) セグメント。例: /path/..;/info

  • 2 つのピリオド (..) で始まるパス (例: /../path/info )

注: URI パスの検証は、照会ストリングには適用されません。

skipEncodedCharVerification プロパティー trueを設定することで、エンコードされた文字の検証をスキップできます。 デフォルト値はfalseです。

サーブレットの要求 URI 内のエンコードされた文字の検証をオプトアウトするには、このプロパティーをアプリケーションの web.xml ファイル内の context-param エレメントとして構成します。

<context-param>
   <param-name>SKIP_ENCODED_CHAR_VERIFICATION</param-name>
   <param-value>true</param-value>
</context-param>

あるいは、これを server.xml ファイル内の webContainer エレメントの属性として設定して、デプロイされているすべてのアプリケーションの検証をスキップすることもできます。

<webContainer skipEncodedCharVerification="true"/>

アプリケーション・レベルの context-param 設定は、サーバー・レベルの webContainer 設定よりも優先されます。

X-Powered-By 応答ヘッダー

デフォルトでは、 X-Powered-By ヘッダーは応答ヘッダーに含まれません。 Servlet 6.0以降では、 X-Powered-By ヘッダーを応答ヘッダーに追加するための構成オプションは使用できません。

削除された API とクラス

以前に非推奨になった以下の API と、それらに対応するコンストラクターおよびメソッドが削除されました。

  • jakarta.servlet.ServletContext - Servlet getServlet(String), Enumeration<Servlet> getServlets(), Enumeration<String> getServletNames(), void log(Exception, String)

  • jakarta.servlet.ServletRequest - String getRealPath(String)

  • jakarta.servlet.ServletRequestWrapper - String getRealPath(String)

  • jakarta.servlet.UnavailableException - UnavailableException(Servlet, String), Servlet getServlet()

  • jakarta.servlet.http.HttpServletRequest - boolean isRequestedSessionIdFromUrl()

  • jakarta.servlet.http.HttpServletRequestWrapper - boolean isRequestedSessionIdFromUrl()

  • jakarta.servlet.http.HttpServletResponse - String encodeUrl(String), String encodeRedirectUrl(String), void setStatus(int, String)

  • jakarta.servlet.http.HttpServletResponseWrapper - String encodeUrl(String)、 String encodeRedirectUrl(String)、v ` oid setStatus(int, String) `

  • jakarta.servlet.http.HttpSession - HttpSessionContext getSessionContext(), Object getValue(String), String[] getValueNames(), void putValue(String, Object), void removeValue(String)

以前に非推奨になった以下のクラスが削除されました。

  • jakarta.servlet.SingleThreadModel

  • jakarta.servlet.http.HttpSessionContext

  • jakarta.servlet.http.HttpUtils

詳しくは、 Jakarta Servlet 6.0 Javadocを参照してください。

Jakarta WebSocket 2.1 と 2.0

Jakarta WebSocket 2.1 フィーチャーは、仕様の API およびメソッドの変更を実装します。 また、ユーザー・プロパティーの処理およびクライアント SSL コンテキストの設定の更新も含まれます。

WsWsocServerContainer#upgrade SPI 用の API の置き換え

Jakarta WebSocket 2.1では、新しい jakarta.websocket.server.ServerContainer#upgradeHttpToWebSocket () API メソッドを使用して、HTTP 要求を WebSocket 接続にプログラマチックにアップグレードすることができます。 このメソッドは、現在非推奨になっている com.ibm.websphere.wsoc.WsWsocServerContainer SPI を、次の WebSocket リリースで削除するプランに置き換えます。 ただし、この SPI は、Jakarta WebSocket フィーチャーのバージョン 2.0 以前でも引き続き使用可能です。

ユーザー・プロパティーの処理の更新

WebSocket 2.1 リリースでは、 ユーザー・プロパティーが変更されています。 サーバー・セッションのユーザー・プロパティーは、最初は ServerEndpointConfig.getUserProperties() メソッドによって設定されます。 このメソッドは、エンドポイントが、着信要求に対してオーバーライドされた ServerEndpointConfig.Configurator#modifyHandshake 呼び出しで設定されている可能性のあるプロパティーを取得できるようにします。

同様に、 クライアント・セッションのユーザー・プロパティー には、 EndpointConfig.getUserProperties() メソッドが取り込まれます。 これらの初期コンテンツはシャロー・コピーです。つまり、 ClientEndpointConfig#getUserProperties および ServerEndpointConfig#getUserProperties() インスタンスは、 WebSocket セッションごとにエンドポイントごとに処理されるようになりました。 プロパティーは、以前の WebSocket バージョンのように、すべてのエンドポイント・インスタンス間でグローバルではなくなりました。

クライアント SSL 構成

WebSocket 2.1以降、アプリケーションは jakarta.websocket.ClientEndpointConfig.Builder#sslContext インスタンスを使用して SSLContext クラスを設定できます。Liberty はこのインスタンスを使用して、サーバーへの wss 接続を確立します。 このオプションは、すべての Liberty サーバーの SSL 構成をオーバーライドするため、注意して使用する必要があります。 「トランスポート・セキュリティー」 機能を有効にして、 server.xml ファイルで SSL 通信を有効にすることをお勧めします。 また、 server.xml ファイルで WAS WebSocket Outbound エレメントを設定することにより、SSL を構成することもできます。

その他の変更点

以下の変更は、 WebSocket 2.1以降で有効です。

  • getter メソッドが、デフォルトのプラットフォーム・コンフィギュレーター jakarta.websocket.server.ServerEndpointConfig.Configurator#getContainerDefaultConfigurator()で使用可能になりました。

  • アプリケーションのデプロイメント中にエンドポイントを追加する必要があるという制限が削除されました。 バージョン 2.1以降では、いつでも新しいエンドポイントを追加できます。

  • jakarta.websocket.Session.getRequestURI() メソッドは、ストリングが存在する場合、プロトコルから照会ストリングへの完全な URI ストリングを戻します。

詳しくは、 Jakarta WebSocket 2.1 の仕様を参照してください。

Jakarta XML Binding 4.0 と 3.0

Jakarta XML Binding 4.0 フィーチャーにより、サード・パーティーの実装に切り替えるオプションが提供され、バリデーター API のサポートが除去されます。

サード・パーティー実装への切り替え

Jakarta XML Binding 4.0で、 jakarta.xml.bind.JAXBContextFactory プロパティーを指定して、優先サード・パーティー実装に切り替えることができます。 これにより、 xmlBinding-4.0 フィーチャーは、参照実装ではなく、指定された実装を使用します。 サード・パーティーの実装を選択する場合は、以下の点を考慮してください。

jakarta.xml.bind.JAXBContextFactory プロパティーを設定した場合は、指定したサード・パーティー実装をアプリケーション・クラスパスに追加する必要があります。 そうでない場合は、 java.lang.ClassNotFoundException 例外が発生します。 さらに、サード・パーティーの実装は、JVM 上で実行されているすべてのアプリケーション間で共有する必要があります。 JVM 上で実行されているアプリケーションのうち、クラスパスに実装が追加されていないアプリケーションでは、 java.lang.ClassNotFoundException 例外が発生します。

バリデーター API を削除します

xmlBinding-4.0 フィーチャーは、 jakarta.xml.bind.Validator API をサポートしていません。 この API を使用するアプリケーションは、それを削除するか、 SchemaValidation インスタンスに置き換える必要があります。 以下の例は、バリデーター API の置換として SchemaValidation インスタンスを使用する方法を示しています。

SchemaFactoryschemaFactory= SchemaFactory.newInstance(XMLConstants.W3C_XML_SCHEMA_NS_URI);
Schema dataObjectSchema= schemaFactory.newSchema(new File("dataObjectSchema.xml")); jaxbUnmarshaller.setSchema(dataObjectSchema)

詳しくは、 Jakarta XML Binding 4.0 specificationを参照してください。
