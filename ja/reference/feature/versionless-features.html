バージョンレス機能で機能選択を簡素化

バージョンレス機能は、Open Liberty 機能のバージョンを指定することなく機能を指定する一般的な方法を提供します。 これらの機能によってバージョン選択が自動化されるため、機能の互換性の問題を心配することなく、アプリケーション開発に集中することができます。

バージョンレス機能は、MicroProfile, Jakarta EE、および Java EE プラットフォームで使用できます。 これらはサーバー環境でのみサポートされており、クライアント環境ではサポートされていません。

バージョンレス機能を有効にする

バージョンレスの機能を有効にするには、対応するバージョン管理されたプラットフォームを宣言する必要がある。 プラットフォームの宣言には3つの選択肢がある：

  • server.xmlファイルのfeatureManager要素内で、platform要素を指定します

  • 環境変数PREFFERED_PLATFORM_VERSIONSを指定する

  • サポートされているプラットフォームから、server.xmlファイルでバージョン管理された機能を有効にする

server.xmlファイルのfeatureManager要素内で、platform要素を指定します

プラットフォームのバージョンを宣言するには、server.xmlファイルのfeatureManager要素の中にplatform要素を指定します。 この方法は、バージョンレス機能を設定・管理する最もシンプルな方法です。

例えば、以下のserver.xmlファイル構成は、jakartaee-9.1のJakarta EEプラットフォームと、servlet、persistence、restfulWSのバージョンレス機能を指定します：

<featureManager>
    <platform>jakartaee-9.1</platform>
    <feature>servlet</feature>
    <feature>jpa</feature>
    <feature>jaxrs</feature>
</featureManager>

この設定により、Open Liberty は各機能を Jakarta EE 9.1 プラットフォームに含まれるバージョンに自動的に解決します。

同様に、以下の例では、MicroProfile 5.0 プラットフォームに含まれているバージョンレスの MicroProfile 機能を有効にします：

<featureManager>
  <platform>microProfile-5.0</platform>
  <feature>mpHealth</feature>
  <feature>mpMetrics</feature>
</featureManager>

環境変数PREFFERED_PLATFORM_VERSIONSを指定する

あるいは、次の例のように、server.env ファイルに PREFERRED_PLATFORM_VERSIONS 環境変数を定義することで、バージョンレス機能のプラットフォームを宣言することもできます：

PREFERRED_PLATFORM_VERSIONS=microProfile-5.0,jakartaee-9.1

この環境変数が設定されていると、server.xml ファイルでバージョンレスの機能を指定することができ、Open Liberty が自動的に正しいバージョンに解決します。 このserver.xmlファイルの例では、mpHealthとmpMetricsの機能は、MicroProfile 5.0:に含まれるバージョンに解決されます

<featureManager>
    <feature>mpHealth</feature>
    <feature>mpMetrics</feature>
</featureManager>

環境シェルからPREFERRED_PLATFORM_VERSIONS環境変数を設定することもできます。 しかし、server package コマンドを実行してサーバーをパッケージ化する必要がある場合、この方法は問題を引き起こすかもしれません。 server.envファイルにPREFERRED_PLATFORM_VERSIONSを設定すると、server packageコマンドを実行したときにサーバーのパッケージングがその設定を保持します。 サーバーイメージを解凍しても、server.envファイルにはPREFERRED_PLATFORM_VERSIONSの設定が残っています。

環境シェルでPREFERRED_PLATFORM_VERSIONS環境変数を設定すると、サーバーがパッケージ化されたときにその変数は保持されません。 このシナリオが発生した場合、server packageコマンドは次のような警告を発します：

CWWKE0969W: A manual PREFERRED_PLATFORM_VERSION environment variable was specified during server packaging.

サーバーイメージを解凍したら、PREFERRED_PLATFORM_VERSIONS変数の定義をサーバーに提供しなければなりません。

server.xmlファイルで、サポートされているプラットフォームのバージョン指定機能を有効にする

サポートされているプラットフォームの1つのバージョンにのみ含まれるバージョン指定機能を少なくとも1つ有効にすることで、暗黙的にプラットフォームを宣言することができます。 Open Liberty は、そのプラットフォームのバージョンレスの機能を、対応するバージョンに自動的に解決します。

次の server.xml の例では、mpHealth-3.0 機能は MicroProfile 4.0 プラットフォームの一部であり、他の MicroProfile プラットフォームにはありません。 Open Liberty は、バージョンレスの mpMetrics および mpConfig 機能を、MicroProfile 4.0: に含まれるバージョンに自動的に解決します

<featureManager>
  <feature>mpHealth-3.0</feature>
  <feature>mpMetrics</feature>
  <feature>mpConfig</feature>
</featureManager>

しかし、MicroProfile 4.0 と MicroProfile 4.1 の両方に含まれている mpConfig-2.0 のように、複数のプラットフォームの一部である機能バージョンを宣言する場合、この戦略は機能しません。 この戦略は、主にいくつかの機能だけを有効にするような単純なコンフィギュレーションに有効である。 同じプラットフォームの一部でありながら、プラットフォームのバージョンが異なる複数のバージョン指定機能を指定した場合、バージョンレス機能が正しく解決されない可能性があります。 より複雑な設定の場合は、platform要素で明示的にプラットフォームを宣言してください。
